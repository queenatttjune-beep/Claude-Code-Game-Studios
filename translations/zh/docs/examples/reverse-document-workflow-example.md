# 反向文档工作流示例

## 场景
开发人员构建了一个可用的技能树系统，但从未编写设计文档。代码存在于 `src/gameplay/skills/` 中，但 `design/gdd/` 中没有 skill-system.md。

---

## 会话记录

**用户：**
```
我在 src/gameplay/skills/ 中有一个可用的技能树系统，大约有 1200 行
代码。我从未编写过设计文档。我们能从实现中创建一个吗？
```

**游戏设计师（读取代码，分析）：**
```
我已经分析了您的技能树实现。以下是我的发现：

结构：
- 3 个技能树：战斗、生存、科技
- 每棵树 5 个层级（通过等级 + 前置条件解锁）
- 共 45 个技能（每棵树 15 个）
- 消耗：技能点（每级 1 点，可花费金币重置）

机制：
- 被动技能（属性加成）
- 主动技能（带冷却的能力）
- 协同加成（第 4+ 层的跨树解锁）

实现细节：
- 数据驱动（assets/data/skills/ 中的 JSON 文件）
- 保存兼容（技能状态已序列化）
- UI 显示锁定/解锁/可用状态

在我起草设计文档之前的问题：

1. **设计意图：** 3 树结构是用于代表玩家角色类型
   （战士/生存者/工程师）还是玩法风格灵活性？

2. **重置消耗：** 重置消耗金币——这是为了：
   - 阻止频繁重置（对构筑的承诺）？
   - 创造金币消耗点（经济平衡）？
   - 还是应该是免费的（鼓励实验）？

3. **协同系统：** 跨树协同出现在第 4 层。这是：
   - 有意的（奖励后期混合构筑）？
   - 还是自然出现的（意外创造了混合元游戏）？

4. **平衡理念：** 我看到有些技能明显更好（第 5 层被动
   是 +50% 伤害）。这是：
   - 力量幻想（后期应该感觉 OP）？
   - 还是不平衡（需要调整）？
```

**用户：**
```
好问题！以下是意图：

1. 是的，3 棵树是角色类型，但我想要灵活性
2. 重置应该便宜——我想要实验
3. 协同是有意的——混合构筑是一个特性
4. 是的，第 5 层应该是强大的，但 +50% 可能太多了
```

**游戏设计师：**
```
完美，这澄清了愿景。让我起草设计文档：

[显示 design/gdd/skill-system.md 草稿]

章节：
1. 概述（3 树系统、混合友好、鼓励重置）
2. 设计支柱：
   - 角色类型清晰度（每棵树有身份）
   - 构筑灵活性（无锁定、廉价重置）
   - 协同奖励（跨树组合可行）
3. 详细设计（树结构、解锁规则、重置）
4. 平衡框架（层级力量曲线、协同限制）
5. 边界情况（战斗中重置？最大技能？上限？）
6. 依赖关系（UI、保存系统、教程）
7. 验收标准（所有树可行、10+ 混合构筑、清晰提示）
8. 未解决问题：
   - 重置成本应该随等级增长还是保持固定？
   - 第 5 层伤害加成：调到 +30% 还是保持 +50%？

**我可以将此写入 design/gdd/skill-system.md 吗？**
```

**用户：**
```
是的，写入。但让我们把第 5 层调到 +30%，并让重置成本随等级增长。
```

**游戏设计师：**
```
✅ 已写入 design/gdd/skill-system.md
✅ 标记为 [从实现反向记录]
✅ 添加了 TODO：将 assets/data/skills/combat-tree.json 中的第 5 层被动更新为 +30%
✅ 添加了 TODO：实现缩放重置成本公式

下一步：
- 运行 /balance-check 来验证技能树进展曲线？
- 创建 ADR 记录为什么是 3 树而不是 5 树？
- 更新教程来解释混合协同？
```

---

## 关键观察

1. **Agent 问了澄清性问题**——不仅仅是描述代码
2. **将意图与实现分开**——用户纠正了"意外"的不平衡
3. **设计文档捕获"为什么"**——不仅仅是"代码做了什么"
4. **识别了缺失的部分**——代码中没有的边界情况
5. **标记了后续工作**——平衡调整、ADR、教程

## 结果

设计文档现在存在，并且**匹配现实**同时也捕获了**预期愿景**。未来的更改将引用此文档，新的团队成员能理解决策背后的*原因*。
