# 示例会话：UX 管线 — /ux-design → /ux-review → /team-ui

**上下文：** 独立开发者，预生产阶段。HUD 和库存屏幕是要设计的首批 UI。游戏有玩家旅程文档和库存及战斗的 GDD。

**起始状态：**
- `design/gdd/inventory-system.md` — 完整
- `design/gdd/combat-system.md` — 完整
- `design/player-journey.md` — 完整（定义了 6 阶段弧线）
- 尚无 UX 规范
- `design/ux/` 目录不存在

**会话时长：** ~2 次会话约 90 分钟（HUD 设计，然后是库存屏幕）
**回合：** 16
**结果：** `design/ux/hud.md` 和 `design/ux/screens/inventory.md` 已创建并审查；team-ui 为视觉实现完成交接

---

## 第 1 部分：使用 /ux-design 进行 HUD 设计

### 第 1 回合：用户开始 HUD 设计

**用户：**
> /ux-design hud

### 第 2 回合：Agent 加载上下文并呈现方案

**Agent（ux-designer，通过 /ux-design）：**
> 开始 HUD 设计。正在加载上下文...
>
> - `design/player-journey.md` 已读取 — 关键状态：平静探索、紧张（检测到威胁）、危机（低 HP/耐力）、战斗
> - `design/gdd/combat-system.md` — HUD 需要：HP 条、耐力条、主动能力冷却（最多 3 个）、伤害数字
> - `design/gdd/inventory-system.md` — HUD 需要：快速栏（4 个栏位）、活动物品指示器
>
> **HUD 哲学问题：** 游戏期间玩家的主要注意力焦点是什么？
>
> 这决定了 HUD 应该有多强势：
> - **选项 A：叙事式（低调）** — HP 仅在受伤害时显示，耐力仅在消耗时显示。世界优先，HUD 除非紧急否则退隐。
> - **选项 B：持久最小化** — HP 和耐力始终可见，小而低调。无需模式切换即可快速参考。
> - **选项 C：全面战术** — 所有信息始终可见（HP、耐力、冷却、地图、库存栏位）。玩家始终完全知晓。
>
> 鉴于您的生存游戏和"孤独幸存者"幻想，A 或 B 与游戏类型最一致。

**用户：**
> B——我希望它始终在那里但最小化。玩家总是在管理资源。

### 第 3 回合：Agent 起草 HUD 哲学部分

**Agent：**
> **HUD 哲学草稿：**
>
> > **方案：持久最小化**
> > HUD 始终可见但占用最小屏幕空间。玩家是管理资源的幸存者——拒绝显示 HP/耐力会造成挫折而非紧张。所有关键状态无需中断探索焦点即可一目了然。元素在变为紧急时（HP 低于 30% 时闪烁、耐力为 0 时闪动）增加视觉重量，然后返回最小状态。
>
> 我可以将此作为哲学部分写入 `design/ux/hud.md` 吗？

**用户：**
> 是的

**Agent 创建 `design/ux/hud.md` 并写入哲学部分。**

*（第 4-7 回合继续逐部分的 HUD 创作：信息架构、区域、元素规范、状态机、视觉预算、平台适配——每个都经起草、批准并写入文件。）*

**所有部分后的最终 HUD 结构：**

```
design/ux/hud.md
  - 哲学：持久最小化
  - 信息架构：HP（左上）、耐力（左上 HP 下方）、快速栏（底部居中）、
    能力冷却（右下）、伤害数字（世界空间）
  - 区域：4 个已定义（左上生命值、底部居中交互、右下能力、世界空间）
  - 元素规范：6 个元素，含尺寸、字号、不透明度规则
  - 状态机：平静 / 紧张 / 危机 / 战斗（4 个状态、过渡触发）
  - 视觉预算：最大 8% 屏幕覆盖、3 个同时动画
  - 平台适配：控制器（添加了瞄准准星）、移动端（v1 范围外）
```

## 第 2 部分：使用 /ux-design 的库存屏幕

### 第 8 回合：用户开始库存屏幕

**用户：**
> /ux-design screen inventory

### 第 9 回合：Agent 扫描 GDD 和玩家旅程

**Agent：**
> 正在加载库存屏幕的上下文...
>
> 来自 `design/gdd/inventory-system.md`：
> - 网格库存，6×8 栏位
> - 物品有重量（影响移动）和堆叠大小
> - 玩家可以装备、丢弃、检查、拆分堆叠
>
> 来自 `design/player-journey.md`：
> - 库存经常在平静探索期间打开
> - 玩家在打开库存时应感到"有条理和掌控"
> - 焦虑时刻："我需要决定丢弃什么"——应该清晰，而非慌乱
>
> **玩家需求（来自旅程）：** *我需要快速评估我携带了什么，决定保留或丢弃什么，然后返回世界。*
>
> **关键设计问题：** 玩家如何导航到库存操作？
> - **选项 A：右键上下文菜单** — 悬停物品 → 右键单击 → 菜单出现（常见/通用）
> - **选项 B：拖拽式** — 拖到丢弃区域丢弃；拖到装备栏装备（Stardew Valley）
> - **选项 C：检查面板** — 单击物品 → 右侧出现详情面板，带操作按钮

**用户：**
> B——拖拽式。我想要触感。

> **裁决：需要修改**
>
> 在移交给 /team-ui 之前必须解决两个问题：
>
> 1. **阻止（可访问性）：** 库存屏幕没有拖拽的键盘替代方案。运动可访问性需要非拖拽的交互路径。添加"按 X 拾取 / 再次按 X 放置"的切换，或带有键盘可访问操作按钮的详情面板。
>
> 2. **建议（HUD）：** HP/耐力条仅使用颜色来传达状态。为色盲用户添加辅助指示器（形状、图标或图案）。可以在视觉通道中解决。
> 3. **建议（库存）：** 满库存的丢弃行为未指定。在实现前在 GDD 中解决，或记录为范围外。

## 此示例展示了什么

- **上下文驱动设计**：agent 读取 player-journey.md 将 HUD 决策建立在玩家情绪状态上，而不是功能清单
- **UX 审查是硬关卡**：`/team-ui` 在进入视觉设计前检查是否有通过的 `/ux-review`
- **可访问性早期捕获**：QA 测试员在最后一周才发现缺失的键盘替代方案——审查在实现前就发现了
- **阻止 vs. 建议**：缺失的键盘路径是 BLOCKING（停止交接）；色盲颜色是 ADVISORY（可以在视觉通道中修复）
- **逐部分的 UX 创作**：与 `/design-system` 相同的增量模式——每个部分在继续前写入文件
- **单独的 HUD 和屏幕文件**：`design/ux/hud.md` 是全局 HUD；按屏幕的规范位于 `design/ux/screens/`
- **/team-ui 强制模式库**：此会话后，库存拖拽成为 `design/ux/interaction-patterns.md` 中供所有未来屏幕引用的已记录模式
