### [会话：使用 /design-system 编写 GDD](session-design-system-skill.md)
**类型：** 设计（skill 驱动）
**Skill：** `/design-system`
**时长：** ~60 分钟（14 回合）
**复杂度：** 中

**场景：**
在 `/map-systems` 生成系统索引后，开发人员运行 `/design-system movement`。Skill 从游戏概念和依赖 GDD 加载上下文，运行技术可行性预检查，然后一次引导所有 8 个 GDD 部分——起草、批准并将每个部分写入磁盘，然后继续下一个。

**关键时刻：**
- 技术可行性预检查标记了 Jolt 物理默认更改（Godot 4.6）
- 增量写入：每个部分在批准后立即写入磁盘
- 在第 5 部分期间会话崩溃 → agent 从第一个空白部分恢复
- 在依赖关系部分期间浮现的依赖信号（耐力、库存）
- 以明确的交接结束："在当前系统之前运行 `/design-review`"

**学习：**
- `/design-system` 与要求 agent "编写 GDD" 有何不同
- 逐部分循环如何防止 30k-token 上下文膨胀
- 增量文件写入如何在会话崩溃后幸存
- Skill 如何浮现下游依赖合约

### [会话：完整 Story 生命周期](session-story-lifecycle.md)
**类型：** 完整工作流
**Skill：** `/story-readiness` → 实现 → `/story-done`
**时长：** ~50 分钟（13 回合）
**复杂度：** 中

**场景：**
开发人员从冲刺积压中拾取一个 story。`/story-readiness` 在任何代码写入之前捕获了滚动方向歧义。实现后，`/story-done` 验证了 9 个验收标准，识别了 2 个延迟标准（库存尚未集成），并附备注关闭了 story。

**关键时刻：**
- `/story-readiness` 在第 2 回合捕获了规范歧义——在实现开始前解决
- ADR 状态检查：如果 ADR 仍为 Proposed，story 会被阻止
- 清单版本检查：确认 story 的指导未与当前架构偏离
- 在无法集成时跟踪（而非丢失）延迟标准
- `sprint-status.yaml` 在 story 关闭时更新，下一个就绪的 story 自动浮现

**学习：**
- 为什么 `/story-readiness` 防止实现后期的歧义
- 延迟标准如何工作（COMPLETE WITH NOTES vs. BLOCKED）
- TR-ID 引用如何防止错误偏差标记
- 从积压 → 实现 → 关闭的完整循环

### [会话：关卡检查和阶段过渡](session-gate-check-phase-transition.md)
**类型：** 阶段关卡
**Skill：** `/gate-check`
**时长：** ~20 分钟（7 回合）
**复杂度：** 低

**场景：**
开发人员完成系统设计阶段并运行 `/gate-check` 以推进。关卡发现所有 6 个 MVP GDD 已完成，交叉审查通过但有一个低严重性顾虑。关卡通过，`stage.txt` 更新，agent 为技术设置提供了具体的有序清单。

**关键时刻：**
- 关卡验证工件存在 AND 内部完整性（每个 GDD 8 个部分）
- CONCERNS ≠ FAIL：低严重性交叉审查备注通过了关卡
- stage.txt 更新改变了 `/help`、`/sprint-status` 和所有后续 skill 看到的内容
- Agent 将交叉审查顾虑作为要编写的具体 ADR 浮现
- 下一阶段清单是具体且有序的，而非泛泛而谈

**学习：**
- 关卡检查实际验证什么（不仅仅是"文件是否存在？"）
- PASS/CONCERNS/FAIL 裁决如何运作
- 为什么 stage.txt 是阶段跟踪的权威
- 阶段过渡后发生了什么变化

### [会话：UX 管线 — /ux-design → /ux-review → /team-ui](session-ux-pipeline.md)
**类型：** UX 设计管线
**Skill：** `/ux-design`、`/ux-review`、`/team-ui`
**时长：** ~90 分钟（16 回合）
**复杂度：** 中高

**场景：**
开发人员设计 HUD 和库存屏幕。`/ux-design` 读取玩家旅程和 GDD 以将决策建立在玩家情绪状态之上。`/ux-review` 捕获了一个阻止性可访问性缺口（无拖放的键盘替代方案）和一个建议性色盲问题。修复后，`/team-ui` 接受交接。

**关键时刻：**
- HUD 哲学选择（叙事式 vs. 持久式 vs. 战术式）基于生存游戏类型惯例
- `/ux-review` 区分 BLOCKING（阻止交接）和 ADVISORY（可在视觉通道中修复）
- 可访问性在实现前捕获，而非 QA 期间
- 键盘替代方案在一轮中添加；审查重新运行并通过
- `/team-ui` 在开始视觉设计前检查是否有通过的 `/ux-review`

**学习：**
- `/ux-design` 如何使用玩家旅程上下文来支撑 UI 决策
- `/ux-review` 实际检查什么（不仅仅是"是否有规范？"）
- HUD 文档（`design/ux/hud.md`）和按屏幕规范的差异
- 可访问性问题在设计时 vs. 实现时如何处理

### [会话：使用 /adopt 的棕地项目过渡](session-adopt-brownfield.md)
**类型：** 棕地采用
**Skill：** `/adopt`
**时长：** ~30 分钟（8 回合）
**复杂度：** 低中

**场景：**
开发人员有 3 个月的现有代码和粗略设计笔记，但格式不对。`/adopt` 审计格式符合性（不仅仅是文件存在），按严重性分类 4 个缺口，构建有序的 7 步迁移计划，并立即修复 BLOCKING 缺口（缺少系统索引），通过从代码库中推断。

**关键时刻：**
- FORMAT 审计区分"文件存在"和"文件具有所需的内部结构"
- BLOCKING 缺口被识别：缺少系统索引阻止 4+ 个 skill 运行
- 迁移计划有序：先阻止性缺口，然后高，然后中
- 系统索引从代码结构引导——棕地代码包含答案
- 改造模式 vs. 新创作：`/design-system retrofit` 填补缺口而不覆盖

**学习：**
- `/adopt` 和 `/project-stage-detect` 之间的区别
- 格式符合性如何检查（章节检测，不仅仅是文件存在）
- 棕地项目如何在不丢失现有工作的情况下过渡
- 何时使用改造模式 vs. 完整创作

### [会话：设计制作系统](session-design-crafting-system.md)
**类型：** 设计
**Agent：** game-designer
**时长：** ~45 分钟（12 回合）
**复杂度：** 中

**场景：**
独立开发人员需要设计一个支撑支柱 2（"通过实验的涌现发现"）的制作系统。Agent 通过问答引导，呈现 3 个带有游戏理论分析的设计选项，整合用户修改，并在每一步获得批准后迭代起草 GDD。

**关键协作时刻：**
- Agent 提前提出 5 个澄清性问题
- 呈现 3 个不同的选项，含优缺点 + MDA 对齐
- 用户修改推荐选项，agent 立即整合
- 边界情况被主动标记（"如果非配方组合会怎样？"）
- 每个 GDD 部分在进入下一部分前展示以获取批准
- 创建文件前明确询问"我可以写入[文件]吗？"

**学习：**
- 设计 agent 如何询问目标、约束、参考
- 如何使用游戏设计理论（MDA、SDT、Bartle）呈现选项
- 如何逐部分迭代草稿
- 何时委派给专业人员（systems-designer、economy-designer）

### [会话：实现战斗伤害计算](session-implement-combat-damage.md)
**类型：** 实现
**Agent：** gameplay-programmer
**时长：** ~30 分钟（10 回合）
**复杂度：** 低中

**场景：**
用户有完整的设计文档并希望实现伤害计算。Agent 读取规范，识别 7 个歧义/缺口，提出澄清性问题，为批准建议架构，在规则执行下实现，并主动编写测试。

**关键协作时刻：**
- Agent 先读取设计文档，识别 7 个规范歧义
- 在实现前展示了包含代码示例的架构方案
- 用户要求类型安全，agent 完善并重新提议
- 规则捕获问题（硬编码值），agent 透明修复
- 按照验证驱动开发主动编写测试
- Agent 提供了下一步选项供选择，而非单方面决定

**学习：**
- 实现 agent 如何在编码前澄清规范
- 如何使用代码示例为批准建议架构
- 规则如何自动执行标准
- 如何处理规范缺口（询问，不要假设）
- 验证驱动开发（测试证明它有效）

### [会话：范围危机 - 战略决策](session-scope-crisis-decision.md)
**类型：** 战略决策
**Agent：** creative-director
**时长：** ~25 分钟（8 回合）
**复杂度：** 高

**场景：**
独立开发人员面临危机：Alpha 里程碑还有 2 周，制作系统需要 3 周，投资者演示是成败关键。创意总监收集上下文，构建决策框架，呈现 3 个带有诚实权衡分析的战略选项，提出推荐但尊重用户决定，然后记录决策并附 ADR 和演示脚本。

**关键协作时刻：**
- Agent 在提出解决方案前读取上下文文档
- 提出 5 个问题以理解决策约束
- 正确构建决策框架（风险是什么、评估标准）
- 呈现 3 个选项，含风险分析和历史先例
- 做出强有力的推荐但明确表示："这是您的决定"
- 记录决策 + 提供演示脚本来支持用户

**学习：**
- 领导力 agent 如何构建战略决策
- 如何呈现带权衡分析的选项
- 如何在推荐中使用游戏开发先例和理论
- 如何记录决策（ADR）
- 如何级联决策到受影响的部门

## 🎯 **这些示例展示的内容**

所有示例遵循**协作工作流模式：**

```
提问 → 选项 → 决策 → 草稿 → 批准
```

> **注意：** 这些示例将协作模式显示为对话文本。
> 在实践中，agent 现在在决策点使用 `AskUserQuestion` 工具
> 来呈现结构化选项选择器（含标签、描述和多选）。
> 模式是**解释 → 捕获**：agent 先在对话中解释分析，
> 然后为用户的决策呈现结构化 UI 选择器。

### ✅ **展示的协作行为：**

1. **Agent 先问，不假设**
   - 设计 agent 询问目标、约束、参考
   - 实现 agent 澄清规范歧义
   - 领导力 agent 在推荐前收集完整上下文

2. **Agent 呈现选项，不是指令**
   - 2-4 个选项带优缺点
   - 基于理论、先例、项目支柱的推理
   - 提出推荐，但由用户决定

3. **Agent 在定稿前展示工作**
   - 设计草稿逐部分展示
   - 架构方案在实现前展示
   - 战略分析在决策前呈现

4. **Agent 在写入文件前获得批准**
   - 在 Write/Edit 工具前明确询问"我可以写入[文件]吗？"
   - 多文件变更先列出所有受影响文件
   - 在任何文件创建前用户说"是"

5. **Agent 根据反馈迭代**
   - 用户修改立即整合
   - 当用户更改推荐时，不防御
   - 庆祝用户改进 agent 的建议

## 🔍 **所有示例中的常见模式**

### 第 1-2 回合：**先理解再行动**
- Agent 读取上下文（设计文档、规范、约束）
- Agent 提出澄清性问题
- 无假设或猜测

### 第 3-5 回合：**呈现带推理的选项**
- 2-4 种不同的方法
- 每个的优缺点
- 支持分析的理论/先例
- 提出推荐，决策权留给用户

### 第 6-8 回合：**迭代草稿**
- 增量展示工作
- 即时整合反馈
- 主动标记边界情况或歧义

### 第 9-10 回合：**批准和完成**
- "我可以写入[文件]吗？"
- 用户："是"
- Agent 写入文件
- Agent 提供下一步（测试、审查、集成）
