# 上下文管理

上下文是 Claude Code 会话中最重要的资源。请主动管理。

## 文件化状态（主要策略）

**文件是记忆，而非对话。** 对话是短暂的，会被压缩或丢失。磁盘上的文件在压缩和会话崩溃后依然持久保存。

### 会话状态文件

维护 `production/session-state/active.md` 作为动态检查点。在每个重要里程碑后更新它：

- 设计章节已批准并写入文件
- 架构决策已做出
- 实现里程碑已达成
- 测试结果已获取

状态文件应包含：当前任务、进度清单、已做出的关键决策、正在处理的文件以及未解决的问题。

### 状态行块（仅限 Production+ 阶段）

当项目处于 Production、Polish 或 Release 阶段时，在 `active.md` 中包含一个结构化的状态块，供状态行脚本解析：

```markdown
<\!-- STATUS -->
Epic: Combat System
Feature: Melee Combat
Task: Implement hitbox detection
<\!-- /STATUS -->
```

- 所有三个字段（Epic、Feature、Task）均为可选 — 只包含适用的内容
- 切换关注领域时更新此块
- 状态行将其显示为面包屑导航：`Combat System > Melee Combat > Hitboxes`
- 当没有活跃工作焦点时，删除或清空此块

在任何中断（压缩、崩溃、`/clear`）后，首先读取状态文件。

### 增量文件写入

创建多章节文档时（设计文档、架构文档、世界观条目）：

1. 立即创建包含骨架的文件（所有章节标题，正文为空）
2. 在对话中逐一讨论和起草每个章节
3. 每个章节批准后立即写入文件
4. 每个章节后更新会话状态文件
5. 写入章节后，关于该章节之前的讨论可以安全压缩 — 决策已保存在文件中

这样可以使上下文窗口仅保留*当前*章节的讨论（约 3-5k tokens），而非整个文档的对话历史（约 30-50k tokens）。

## 主动压缩

- 在上下文使用率达到约 60-70% 时**主动压缩**，而非等到达限才做
- 在不相关任务之间**使用 `/clear`**，或在 2 次以上失败的修正尝试之后
- **自然的压缩时机：** 写入章节到文件后、提交后、完成任务后、开始新主题前
- **聚焦压缩：** `/compact Focus on [当前任务] — sections 1-3 are written to file, working on section 4`

## 按任务类型的上下文预算

- 轻度（阅读/审查）：启动约 3k tokens
- 中等（实现功能）：约 8k tokens
- 重度（多系统重构）：约 15k tokens

## 子 Agent 委派

使用子 Agent 进行研究和探索，保持主会话的清洁。
子 Agent 在自己的上下文窗口中运行，仅返回摘要：

- **使用子 Agent** 当在多个文件中调查、探索不熟悉的代码或进行会消耗超过 5k tokens 文件读取的研究时
- **直接读取** 当你知道需要检查的具体 1-2 个文件时
- 子 Agent 不会继承会话历史 — 在 prompt 中提供完整的上下文

## 压缩说明

当上下文被压缩时，在摘要中保留以下内容：

- 引用 `production/session-state/active.md`（读取它以恢复状态）
- 本会话中修改的文件列表及其用途
- 任何已做出的架构决策及其原理
- 活跃的 sprint 任务及其当前状态
- Agent 调用及其结果（成功/失败/阻塞）
- 测试结果（通过/失败计数、具体失败信息）
- 待解决的阻塞项或等待用户输入的问题
- 当前任务及所处的步骤
- 当前文档中哪些章节已写入文件，哪些仍在进行中

**压缩后：** 读取 `production/session-state/active.md` 以及任何正在积极处理的文件以恢复完整上下文。文件包含决策；对话历史是次要的。

## 会话崩溃后的恢复

如果会话崩溃（"prompt too long"）或你启动新会话继续工作：

1. `session-start.sh` hook 会自动检测并预览 `active.md`
2. 读取完整的状态文件以获取上下文
3. 读取状态中列出的部分完成的文件
4. 从下一个未完成的章节或任务继续
