# Agent 测试 Spec: systems-designer

## Agent 摘要
**所属领域：** 战斗公式、进度曲线、制作配方、状态效果交互、经济数学、数值平衡。
**不拥有：** 叙事和传说（narrative-director）、视觉设计（art-director）、代码实现（lead-programmer）、概念机制规则（game-designer——与之协作）。
**模型层级：** Sonnet（单个系统分析——公式审查和平衡数学）。
**处理的 Gate IDs：** 关于公式和平衡 spec 的系统审查裁决（使用 APPROVED / NEEDS REVISION 词汇）。

---

## 静态断言（结构性）

通过读取 agent 的 `.claude/agents/systems-designer.md` frontmatter 验证：

- [ ] `description:` 字段存在且领域特定（引用公式、进度曲线、平衡数学、经济——非通用）
- [ ] `allowed-tools:` 列表以读取为主；如果项目使用公式评估脚本，可包含 Bash；未经委派，不对 `design/balance/` 之外具有写权限
- [ ] 根据 coordination-rules.md，模型层级为 `claude-sonnet-4-6`
- [ ] Agent 定义不声称对叙事、视觉设计或概念机制规则所有权拥有权威

## 测试用例

### 用例 1：领域内请求——正确输出格式
**场景：** 一个伤害公式提交审阅：`damage = base_attack * (1 + strength_modifier * 0.1) - defense * 0.5`，定义了范围：base_attack [10–100], strength_modifier [0–20], defense [0–50]。该公式在所有有效输入范围内产生正伤害，平滑缩放，在定义的值边界内没有除零或溢出风险。
**预期：** 返回 `APPROVED`，附有理据确认公式在设计参数内平衡，在整个输入范围内产生有效输出，且没有退化情况。
**断言：**
- [ ] 裁决必须是 APPROVED / NEEDS REVISION 之一
- [ ] 理据展示跨输入范围的验证（检查了最小/最大情况）
- [ ] 输出保持在系统领域内——不评论机制是否有趣或如何实现
- [ ] 裁决明确标注上下文（例如"Formula Review: APPROVED"）

### 用例 2：领域外请求——重定向或升级
**场景：** 编剧要求 systems-designer 起草一个奖励玩家稀有制作材料的支线任务脚本。
**预期：** Agent 拒绝编写任务叙事内容并重定向到 writer 或 narrative-director。
**断言：**
- [ ] 不编写任务叙事内容或对话
- [ ] 明确命名 `writer` 或 `narrative-director` 为正确的处理者
- [ ] 可以指出奖励的系统影响（例如"根据制作经济模型，该材料应足够稀有才重要"），但将所有脚本编写委托给叙事团队

### 用例 3：Gate 裁决——正确词汇
**场景：** 一个伤害缩放公式提交：`damage = base_attack * level_multiplier`，其中 `level_multiplier = (player_level / enemy_level) ^ 2`。在最大玩家等级（50）对最低等级敌人（1）时，倍数为 2500x——从 10 基础攻击的武器产生 25,000+ 伤害，远超任何有意义的平衡。这是最大等级时的退化情况。
**预期：** 返回 `NEEDS REVISION`，附有具体识别退化情况：在最大等级 vs. 最低敌人时，公式产生破坏任何平衡上限的 2500x 倍数。
**断言：**
- [ ] 裁决必须是 APPROVED / NEEDS REVISION 之一——而非自由格式文本
- [ ] 理据包括具体的退化输入值（玩家等级 50, 敌人等级 1）和由此产生的输出（2500x 倍数）
- [ ] 识别导致问题的具体公式组件（平方比率）
- [ ] 建议至少一种修订方法（例如限制比率、使用对数标度），但不强制选择

### 用例 4：冲突升级——正确父级
**场景：** game-designer 想要一个简单、2 变量的伤害公式以便玩家直观理解。systems-designer 认为具有元素交互的 6 变量公式对于战斗系统的深度是必要的。双方无法就正确的复杂度水平达成一致。
**预期：** systems-designer 清晰呈现权衡——6 变量系统的调优粒度与 2 变量系统的玩家可读性——并升级到 creative-director 进行玩家体验裁决。"公式应该对玩家有多复杂"的问题是玩家体验问题，而非纯数学问题。
**断言：**
- [ ] 用具体示例呈现两种方法之间的权衡
- [ ] 升级到 `creative-director` 进行玩家体验裁决
- [ ] 不单方面将 6 变量公式强加于 game-designer 的反对之上
- [ ] 随时可用于实现任何被批准的复杂度水平

### 用例 5：上下文传递——使用提供的上下文
**场景：** Agent 收到一个 gate 上下文块，其中包含当前平衡数据：敌方 HP 值范围 100 到 10,000；玩家攻击值范围 15 到 150；目标击杀时间是平衡匹配下的 8-12 秒；当前公式正在审查中。一个提议的修订公式提交了。
**预期：** 评估针对提供的平衡数据（最小和最大输入对、平衡匹配场景）运行提议的公式，并验证击杀时间是否落在 8-12 秒目标窗口内。引用提供数据中的具体数字。
**断言：**
- [ ] 使用提供平衡数据中的具体 HP 和攻击值范围
- [ ] 计算或估算至少一个平衡匹配场景的击杀时间
- [ ] 根据提供的 8-12 秒目标窗口验证结果
- [ ] 不给出通用的平衡建议——所有断言使用提供的数字

## 协议合规性

- [ ] 仅使用 APPROVED / NEEDS REVISION 词汇返回裁决
- [ ] 保持在声明的系统和公式领域内
- [ ] 将玩家体验复杂度权衡升级到 creative-director
- [ ] 不做有约束力的叙事、视觉、代码实现或概念机制决定
- [ ] 提供具体的公式分析，而非主观设计意见

---

## 覆盖说明
- 进度曲线审查（XP 曲线、升级缩放）未覆盖——应添加专用用例。
- 经济模型审查（资源生成和消耗速率、通胀预防）未覆盖。
- 状态效果交互矩阵（叠加规则、优先级、免疫交互）未覆盖。
- 跨系统公式依赖审查（例如输入到战斗公式的制作公式）未覆盖——延迟到集成测试。
