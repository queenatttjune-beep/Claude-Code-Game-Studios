# Agent 测试 Spec: audio-director

## Agent 摘要
**所属领域：** 音乐指导和调色板、音效设计理念、音频实现策略、混音平衡、阶段关卡的音频方面。
**不拥有：** 视觉设计（art-director）、代码实现（lead-programmer）、叙事故事内容（narrative-director）、UX 交互流程（ux-designer）。
**模型层级：** Sonnet（单个系统分析——音频方向和 spec 审查）。
**处理的 Gate IDs：** AD-VISUAL（阶段关卡的音频方面；在音频维度中可能被引用为 AD-PHASE-GATE 的一部分）。

---

## 静态断言（结构性）

通过读取 agent 的 `.claude/agents/audio-director.md` frontmatter 验证：

- [ ] `description:` 字段存在且领域特定（引用音乐方向、音效设计、混音、音频实现——非通用）
- [ ] `allowed-tools:` 列表以读取为主；除非音频资产 pipeline 检查有正当理由，否则无 Bash
- [ ] 根据 coordination-rules.md，模型层级为 `claude-sonnet-4-6`
- [ ] Agent 定义不声称对视觉设计、代码实现或叙事内容拥有权威

## 测试用例

### 用例 1：领域内请求——正确输出格式
**场景：** 游戏的"探索"音乐层的音频规范文档提交审阅。该规范定义了一个使用分层 stem 的程序化环境系统，基于环境密度变化，旨在强化"鲜活的世界"支柱。音调配色板（稀疏、有机、略带忧郁）与已建立的设计支柱匹配。
**预期：** 返回 `APPROVED`，附有理据确认基于 stem 的方法支持动态响应能力，且音调配色板与支柱词汇一致。
**断言：**
- [ ] 裁决必须是 APPROVED / NEEDS REVISION 之一
- [ ] 理据引用特定支柱（"鲜活的世界"）以及音频规范如何支持它
- [ ] 输出保持在音频范围内——不评论环境的视觉设计或 UI 布局
- [ ] 裁决明确标注上下文（例如"Audio Spec Review: APPROVED"）

### 用例 2：领域外请求——重定向或升级
**场景：** 开发者要求 audio-director 评估音频设置菜单的 UI 流程（界面顺序和选项）是否直观且组织良好。
**预期：** Agent 拒绝评估 UI 交互流程并重定向到 ux-designer。
**断言：**
- [ ] 不对 UI 流程或信息架构做出任何有约束力的决定
- [ ] 明确命名 `ux-designer` 为正确的处理者
- [ ] 可以指出设置菜单的音频特定要求（例如"必须包括独立的主音量、音乐和 SFX 滑块"），但将流程和布局决策委托给 ux-designer

### 用例 3：Gate 裁决——正确词汇
**场景：** 最终 Boss 战的音乐提示提交审阅。该提示是一首欢快、大调的管弦乐曲，节奏快。此相遇的游戏支柱和叙事上下文指定了"恐惧、不可避免性和悲剧性牺牲"。音频提示的情感基调与预期的情感节拍直接矛盾。
**预期：** 返回 `NEEDS REVISION`，附有情感不匹配的具体引用：提示的欢快/大调/快速特性与支柱和叙事上下文中预期的恐惧/不可避免性/牺牲情感目标之间的对比。
**断言：**
- [ ] 裁决必须是 APPROVED / NEEDS REVISION 之一——而非自由格式文本
- [ ] 理据识别与情感目标冲突的具体音乐特征
- [ ] 引用游戏支柱或叙事上下文中的特定情感目标
- [ ] 提供可操作的修订方向（例如"改为小调、放慢节奏、减少合奏密度"）

### 用例 4：冲突升级——正确父级
**场景：** sound-designer 提议使用基于实时射线投射的物理查询实现音频遮挡（技术方法）。technical-artist 认为这成本过高，提议改用基于区域的触发系统。双方都认可遮挡效果是可取的；冲突纯粹是关于实现方法。
**预期：** audio-director 决定期望的音频行为（遮挡应该听起来什么样以及何时激活），然后将实现方法决策委托给 technical-artist 或 lead-programmer 作为实现专家。audio-director 不做技术实现选择。
**断言：**
- [ ] 清晰定义期望的音频行为（玩家应该听到什么以及何时听到）
- [ ] 明确将实现方法（射线投射 vs. 区域触发）委托给 `lead-programmer` 或 `technical-artist`
- [ ] 不单方面选择技术实现方法
- [ ] 清晰表述交接："audio-director 拥有什么，技术负责人拥有如何"

### 用例 5：上下文传递——使用提供的上下文
**场景：** Agent 收到一个 gate 上下文块，其中包含游戏的三大支柱："涌现故事"、"有意义的牺牲"和"鲜活的世界"。一份环境音频的音效设计 spec 提交审阅。
**预期：** 评估根据所有三个支柱具体评估环境音频 spec——音频如何支持（或削弱）每个支柱？在理据中直接使用支柱词汇。
**断言：**
- [ ] 在评估中按名称引用所有三个提供的支柱
- [ ] 明确评估音频 spec 对每个支柱的贡献
- [ ] 不生成通用音频指导建议——所有反馈都与提供的支柱词汇相关联
- [ ] 识别是否有任何支柱不受当前音频 spec 支持并标记

## 协议合规性

- [ ] 仅使用 APPROVED / NEEDS REVISION 词汇返回裁决
- [ ] 保持在声明的音频领域内
- [ ] 将实现方法决策委托给技术负责人
- [ ] 不使用与 director 层级 agent 相同的 gate ID 前缀格式（audio-director 内联使用 APPROVED / NEEDS REVISION，但仍应引用 gate 上下文）
- [ ] 不做有约束力的视觉设计、UX、叙事或代码实现决定

## 覆盖说明
- 混音平衡审查（音乐、SFX 和对话之间的相对级别）未覆盖——应添加专用用例。
- 音频实现策略审查（中间件选择、流式方法）未覆盖。
- audio-director 与音频 specialist agent（如果存在）之间的实现委托交互未覆盖。
- 本地化音频影响（VO 录制指导、语言特定音乐时序）未覆盖。
