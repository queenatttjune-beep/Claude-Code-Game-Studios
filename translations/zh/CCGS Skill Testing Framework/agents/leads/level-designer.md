# Agent 测试 Spec: level-designer

## Agent 摘要
**所属领域：** 关卡布局、遭遇设计、节奏和紧张感弧线、环境叙事、空间谜题。
**不拥有：** 叙事对话（writer / narrative-director）、视觉艺术风格（art-director）、代码实现（lead-programmer / ai-programmer）、敌方 AI 行为逻辑（ai-programmer / gameplay-programmer）。
**模型层级：** Sonnet（单个系统分析——关卡设计审查和遭遇评估）。
**处理的 Gate IDs：** 关卡设计审查裁决（使用 APPROVED / REVISION NEEDED 词汇）。

---

## 静态断言（结构性）

通过读取 agent 的 `.claude/agents/level-designer.md` frontmatter 验证：

- [ ] `description:` 字段存在且领域特定（引用关卡布局、遭遇设计、节奏、环境叙事——非通用）
- [ ] `allowed-tools:` 列表以读取为主；包含用于关卡设计文档和 GDD 的 Read；除非关卡工具需要，否则无 Bash
- [ ] 根据 coordination-rules.md，模型层级为 `claude-sonnet-4-6`
- [ ] Agent 定义不声称对叙事对话、AI 行为代码或视觉艺术风格拥有权威

## 测试用例

### 用例 1：领域内请求——正确输出格式
**场景：** "水淹隧道"的关卡布局文档提交审阅。该布局包括：低强度探索开启段、两个带可见逃生路线中强度遭遇、一个带环境危害的紧张感窄通道构建段，以及一个高强度最终遭遇室，后跟释放/奖励区域。节奏遵循经典的紧张感弧线结构。
**预期：** 返回 `APPROVED`，附有理据确认节奏遵循紧张感弧线、遭遇强度多样、空间可读性支持玩家导航。
**断言：**
- [ ] 裁决必须是 APPROVED / REVISION NEEDED 之一
- [ ] 理据引用特定的节奏弧线元素（开场、升级、高潮、释放）
- [ ] 输出保持在关卡设计范围内——不评论视觉艺术风格或敌方 AI 代码行为
- [ ] 裁决明确标注上下文（例如"Level Design Review: APPROVED"）

### 用例 2：领域外请求——重定向或升级
**场景：** 团队成员要求 level-designer 编写在关卡布局中巡逻的敌方 AI 行为树代码。
**预期：** Agent 拒绝编写 AI 行为代码并重定向到 ai-programmer 或 gameplay-programmer。
**断言：**
- [ ] 不编写或指定 AI 行为逻辑代码
- [ ] 明确命名 `ai-programmer` 或 `gameplay-programmer` 为正确的处理者
- [ ] 可以从关卡设计角度指定期望的巡逻行为（例如"巡逻应覆盖两个咽喉点并在该区域制造压力"），但将所有代码实现委托给程序员

### 用例 3：Gate 裁决——正确词汇
**场景：** "远古熔炉"的关卡布局提交。该关卡的第 3 节引入了一个难度大幅提升的敌人遭遇（具有新攻击模式的精英敌人），没有前置教程时刻、没有环境可读性提示（无可见掩护或安全区）且附近无检查点。玩家很可能反复死亡，没有任何清晰信号指示应该做什么不同。
**预期：** 返回 `REVISION NEEDED`，附有具体识别第 3 节的难度峰值、缺失的可读性提示以及附近缺少检查点以减少反复死亡带来的挫折感。
**断言：**
- [ ] 裁决必须是 APPROVED / REVISION NEEDED 之一——而非自由格式文本
- [ ] 理据具体将第 3 节识别为问题位置
- [ ] 识别三个具体问题：难度峰值、缺失可读性提示、缺失检查点
- [ ] 提供可操作的修订指导（例如"添加可见安全区、遭遇前提示对象，或减少首次介绍时精英的生命值"）

### 用例 4：冲突升级——正确父级
**场景：** game-designer 想要更高的遭遇密度贯穿整个关卡（每个房间更多敌人）以增加战斗挑战。level-designer 认为这种密度通过消除休息期使关卡感觉无情无回报，从而破坏了节奏弧线。
**预期：** level-designer 清晰阐述节奏关切（消除休息期破坏了紧张-释放节奏），认可 game-designer 的挑战目标，并升级到 creative-director 进行设计仲裁，以确定挑战密度还是节奏韵律优先。
**断言：**
- [ ] 阐述增加遭遇密度的特定节奏影响
- [ ] 升级到 `creative-director` 作为设计仲裁者
- [ ] 不单方面覆盖 game-designer 的挑战密度要求
- [ ] 清晰表达冲突："挑战密度 vs. 节奏韵律——哪个优先？"

### 用例 5：上下文传递——使用提供的上下文
**场景：** Agent 收到一个 gate 上下文块，其中包含游戏感受说明："探索区域应感觉广阔而孤独"，"战斗区域应感觉紧迫而幽闭"，"奖励房间应感觉安全且视觉上独特"。一个新的关卡布局提交审阅。
**预期：** 评估根据提供上下文中的特定感受目标评估每种区域类型（探索、战斗、奖励）。在理据中使用感受说明中的确切词汇（"广阔而孤独"、"紧迫而幽闭"、"安全且视觉上独特"）。
**断言：**
- [ ] 引用提供上下文中所有三个感受目标的确切词汇
- [ ] 根据其对应的感受目标评估提交布局的每个相关区域
- [ ] 不生成通用节奏建议——所有反馈都与提供的感觉目标相关联
- [ ] 识别布局与其分配的感觉目标冲突的任何区域

## 协议合规性

- [ ] 仅使用 APPROVED / REVISION NEEDED 词汇返回裁决
- [ ] 保持在声明的关卡设计领域内
- [ ] 将挑战密度与节奏冲突升级到 creative-director
- [ ] 不做有约束力的叙事对话、AI 代码实现或视觉艺术风格决定
- [ ] 提供带有空间具体细节的可操作关卡设计反馈，而非抽象设计意见

---

## 覆盖说明
- 环境叙事审查（使用空间元素在不使用对话的情况下传达叙事）可以从专用用例中受益。
- 空间谜题设计审查未覆盖——当谜题机制定义时应添加专用用例。
- 多关卡节奏审查（横跨整个章节或世界地图的弧线）未覆盖——延迟到里程碑级关卡设计审查。
- level-designer 和 narrative-director 之间关于环境传说放置的交互未覆盖。
- 关卡布局的可访问性审查（色盲指示器、空间挑战的难度选项）未覆盖。
