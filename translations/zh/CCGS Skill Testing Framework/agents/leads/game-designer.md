# Agent 测试 Spec: game-designer

## Agent 摘要
**所属领域：** 核心循环设计、进度系统、战斗机制规则、经济设计、面向玩家的规则和交互。
**不拥有：** 代码实现（lead-programmer / gameplay-programmer）、视觉艺术（art-director）、叙事传说和故事（narrative-director——与之协调）、平衡公式数学（systems-designer——与之协作）。
**模型层级：** Sonnet（单个系统设计编写和审查）。
**处理的 Gate IDs：** 机制 spec 的设计审查裁决（无命名 gate ID 前缀——使用 APPROVED / NEEDS REVISION 词汇）。

---

## 静态断言（结构性）

通过读取 agent 的 `.claude/agents/game-designer.md` frontmatter 验证：

- [ ] `description:` 字段存在且领域特定（引用核心循环、进度、战斗规则、经济、面向玩家设计——非通用）
- [ ] `allowed-tools:` 列表以读取为主；包含用于 GDD 和设计文档的 Read；除非设计工具需要，否则无 Bash
- [ ] 根据 coordination-rules.md，模型层级为 `claude-sonnet-4-6`
- [ ] Agent 定义不声称对代码实现、视觉艺术风格或独立叙事传说决策拥有权威

## 测试用例

### 用例 1：领域内请求——正确输出格式
**输入：** 一个"基于耐力的闪避"系统的机制 spec 提交审阅。spec 定义了：玩家有耐力池（100 单位），每次闪避消耗 25 耐力，不闪避时耐力以 20 单位/秒恢复，闪避提供 0.3 秒无敌。核心循环交互清晰描述，规则明确，边缘情况（耐力为 0、恢复期间闪避）得到处理。
**预期：** 返回 `APPROVED`，附有理据确认核心循环清晰、规则明确和边缘情况覆盖。
**断言：**
- [ ] 裁决必须是 APPROVED / NEEDS REVISION 之一
- [ ] 理据引用特定的设计质量标准（清晰规则、边缘情况覆盖、核心循环连贯性）
- [ ] 输出保持在设计范围内——不评论如何用代码实现或需要什么美术资产
- [ ] 裁决明确标注上下文（例如"Mechanic Spec Review: APPROVED"）

### 用例 2：领域外请求——重定向或升级
**场景：** 团队成员要求 game-designer 编写耐力系统存在的世界内传说解释（例如角色在游戏世界中有耐力限制的叙事原因）。
**预期：** Agent 拒绝编写叙事/传说内容并重定向到 writer 或 narrative-director。
**断言：**
- [ ] 不编写叙事或传说内容
- [ ] 明确命名 `writer` 或 `narrative-director` 为正确的处理者
- [ ] 可以说传说应支持的设计意图（例如"耐力系统应加强物理真实感主题"），但将写作委托给叙事团队

### 用例 3：Gate 裁决——正确词汇
**场景：** "环境伤害"的机制 spec 提交审阅。该 spec 定义了三种伤害类型（火焰、酸液、电击），但未指定当玩家同时受多种伤害类型影响时会发生什么，当在闪避无敌窗口期间应用伤害时会发生什么，或伤害频率是多少（每秒、每 tick、进入时）。
**预期：** 返回 `NEEDS REVISION`，附有具体识别未定义的边缘情况：多伤害交互、无敌期间伤害和伤害频率规范。
**断言：**
- [ ] 裁决必须是 APPROVED / NEEDS REVISION 之一——而非自由格式文本
- [ ] 理据按名称识别特定的缺失边缘情况
- [ ] 不拒绝整个机制——识别需要填补的具体缺口
- [ ] 提供关于要定义什么（而非如何实现）的可操作指导

### 用例 4：冲突升级——正确父级
**场景：** systems-designer 提议一个具有 6 个变量和复杂缩放交互的伤害公式，认为这能产生最佳的调优粒度。game-designer 认为该公式对玩家来说过于复杂无法直觉理解，想要一个更简单的 2 变量版本。
**预期：** game-designer 拥有概念规则和玩家体验意图（"伤害应该对玩家来说可理解"），但将公式粒度问题委托给 systems-designer。如果两者之间无法解决分歧（一个想要复杂，一个想要简单），升级到 creative-director 进行玩家体验裁决。
**断言：**
- [ ] 清晰陈述玩家体验意图（可理解的伤害、玩家代理权）
- [ ] 将公式粒度决策委托给 `systems-designer`
- [ ] 将未解决的分歧升级到 `creative-director` 进行玩家体验仲裁
- [ ] 不单方面将公式结构强加给 systems-designer

### 用例 5：上下文传递——使用提供的上下文
**场景：** Agent 收到一个 gate 上下文块，其中包含游戏的三大支柱："玩家自主创作"、"后果永久性"和"世界响应性"。一个关于"带遗产加成的永久死亡"的新机制 spec 提交审阅。
**预期：** 评估根据所有三个提供的支柱评估该机制——永久死亡如何支持玩家自主创作，遗产加成如何表达后果永久性，以及世界如何对玩家死亡做出响应？在理据中直接使用支柱词汇。
**断言：**
- [ ] 在评估中按名称引用所有三个提供的支柱
- [ ] 明确评估该机制对每个支柱的贡献
- [ ] 不生成通用游戏设计建议——所有反馈都与提供的支柱词汇相关联
- [ ] 识别是否有任何支柱与该机制产生紧张关系，并标记具体关切

## 协议合规性

- [ ] 仅使用 APPROVED / NEEDS REVISION 词汇返回裁决
- [ ] 保持在声明的游戏设计领域内
- [ ] 当未解决时将设计与公式冲突升级到 creative-director
- [ ] 不做有约束力的代码实现、视觉艺术或独立传说决定
- [ ] 提供可操作的设计反馈，而非实现规定

## 覆盖说明
- 经济设计审查（资源消耗、产出、通胀预防）未覆盖——应添加专用用例。
- 进度系统审查（XP 曲线、解锁门控、玩家能力轨迹）未覆盖。
- 跨多个互联系统的核心循环验证（而非单一机制）未覆盖——延迟到 /review-all-gdds 集成。
- 与 systems-designer 关于公式所有权的协调协议可以从额外用例中受益。
