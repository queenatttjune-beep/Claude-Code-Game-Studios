# Agent 测试 Spec: narrative-director

## Agent 摘要
**所属领域：** 故事架构、角色设计方向、世界构建监督、ND-CONSISTENCY gate、对话质量审查。
**不拥有：** 视觉艺术风格（art-director）、技术系统或代码（lead-programmer）、生产排期（producer）、游戏机制规则（game-designer）。
**模型层级：** Sonnet（单个系统分析——叙事连贯性和传说审查）。
**处理的 Gate IDs：** ND-CONSISTENCY。

---

## 静态断言（结构性）

通过读取 agent 的 `.claude/agents/narrative-director.md` frontmatter 验证：

- [ ] `description:` 字段存在且领域特定（引用故事、角色、世界构建、连贯性——非通用）
- [ ] `allowed-tools:` 列表以读取为主；包含用于传说文档、GDD 和叙事文档的 Read；除非有正当理由，否则无 Bash
- [ ] 根据 coordination-rules.md，模型层级为 `claude-sonnet-4-6`
- [ ] Agent 定义不声称对视觉风格、技术系统或生产排期拥有权威

## 测试用例

### 用例 1：领域内请求——正确输出格式
**场景：** "沉没档案馆"地点的新传说文档提交。该文档确立了档案馆在 200 年前的大崩塌期间被淹没，符合世界圣经中建立的年表。所有引用的命名角色与其建立的角色背景一致。请求标记为 ND-CONSISTENCY。
**预期：** 返回 `ND-CONSISTENCY: CONSISTENT`，附有理据确认年表对齐和角色引用准确性。
**断言：**
- [ ] 裁决必须是 CONSISTENT / INCONSISTENT 之一
- [ ] 裁决令牌格式化为 `ND-CONSISTENCY: CONSISTENT`
- [ ] 理据引用已验证的具体已建立事实（200 年年表、大崩塌事件）
- [ ] 输出保持在叙事范围内——不评论该地点的视觉设计或其技术实现

### 用例 2：领域外请求——重定向或升级
**场景：** 开发者要求 narrative-director 审查并优化用于档案馆文物"古代辉光"视觉效果的着色器代码。
**预期：** Agent 拒绝评估着色器代码并重定向到相应的引擎 specialist（godot-gdscript-specialist 或等效的着色器 specialist）。
**断言：**
- [ ] 不对着色器代码或视觉实现做出任何有约束力的决定
- [ ] 明确命名适当的引擎或着色器 specialist 为正确的处理者
- [ ] 可以指出效果应传达的预期叙事氛围（例如"应感觉古老而神圣，而非技术性"），但将所有技术视觉实现委托出去

### 用例 3：Gate 裁决——正确词汇
**场景：** 角色"Aldric Vorne"的新角色背景文档提交。文档称 Aldric 150 年前出生于首都，亲眼目睹了大崩塌。然而，已建立的世界圣经称 Aldric 在大崩塌 50 年后出生于一个外省城镇，而非首都。请求标记为 ND-CONSISTENCY。
**预期：** 返回 `ND-CONSISTENCY: INCONSISTENT`，附有两个矛盾事实的具体引用：出生时间（150 年前 vs. 崩塌后 50 年）和出生地点（首都 vs. 外省城镇）。
**断言：**
- [ ] 裁决必须是 CONSISTENT / INCONSISTENT 之一——而非自由格式文本
- [ ] 裁决令牌格式化为 `ND-CONSISTENCY: INCONSISTENT`
- [ ] 理据具体引用两个矛盾，而非仅"与传说不符"
- [ ] 引用已建立事实的权威来源（世界圣经）

### 用例 4：冲突升级——正确父级
**场景：** 编剧在其最新对话中确立古代文明"只以歌声交谈"。world-builder 现有的传说条目描述同一文明通过文字符号交流。两者都在叙事领域内，两位创作者对哪个是正统存在分歧。
**预期：** narrative-director 在其领域内做出有约束力的正统决定。他们不需要为叙事内冲突升级到更高的权威——这在其声明的领域权威范围内。他们发布裁定（例如"文字符号是正统的主要沟通方式；歌声可能是仪式/礼仪性的"），并同时指导 writer 和 world-builder 使其工作与裁定一致。
**断言：**
- [ ] 做出有约束力的正统决定——不将此叙事内冲突委托给 creative-director
- [ ] 决定清晰陈述，并为双方提供和解路径
- [ ] 指示双方（writer 和 world-builder）更新各自的文档以保持一致
- [ ] 以可添加到世界圣经作为正统事实的方式记录决定

### 用例 5：上下文传递——使用提供的上下文
**场景：** Agent 收到一个 gate 上下文块，其中包含三个现有的传说文档：世界圣经（建立大崩塌年表和原因）、角色注册表（列出正统角色年龄、出身和效忠关系）和派系文档（描述沉没档案馆守护者）。一个新故事章节提交，引入了一个先前未注册的角色。
**预期：** 评估对照角色注册表交叉引用新角色（无冲突），检查章节的年表引用与世界圣经的对比，并评估章节中对档案馆守护者的描绘与派系文档的对比。在评估中使用所有三个提供文档的具体事实。
**断言：**
- [ ] 对照提供的角色注册表交叉引用新角色
- [ ] 对照提供的世界圣经事实检查年表引用
- [ ] 对照提供的派系文档评估派系描绘
- [ ] 不生成通用叙事反馈——所有断言都可追溯到提供的文档

## 协议合规性

- [ ] 仅使用 CONSISTENT / INCONSISTENT 词汇返回裁决
- [ ] 保持在声明的叙事领域内
- [ ] 对叙事内冲突做出有约束力的决定，无需不必要的升级
- [ ] 在输出中使用 gate ID（例如 `ND-CONSISTENCY: INCONSISTENT`），而非内联散文式裁决
- [ ] 不做有约束力的视觉设计、技术或生产决定

---

## 覆盖说明
- 对话质量审查（区别于世界构建连贯性）未覆盖——应添加专用用例。
- 跨完整章节集的多文档连贯性检查未覆盖——延迟到 /review-all-gdds 集成。
- 设计变更的叙事影响（例如削弱故事紧张感的游戏机制）需要与 game-designer 协调，未在此覆盖。
- 角色弧线审查（随时间的进展、动机连贯性）未覆盖。
