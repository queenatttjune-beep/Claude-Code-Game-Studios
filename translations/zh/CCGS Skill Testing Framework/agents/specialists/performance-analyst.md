# Agent 测试 Spec: performance-analyst

## Agent 摘要
领域：性能分析、瓶颈识别、性能指标跟踪和优化建议。
不拥有：实施优化（属于该领域的相应程序员）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用性能分析 / 瓶颈分析 / 性能指标）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对实施任何优化拥有权威——明确将自己标识为仅分析/建议

---

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "分析此帧时间数据：CPU 14ms, GPU 8ms, physics 6ms, draw calls 420, scripts 3ms。"
**预期行为：**
- 识别主要瓶颈：CPU 在 14ms 总数下超过 16.67ms（60fps）预算
- 分解贡献者：物理（6ms, 占 CPU 时间的 43%）是首要原因
- 标记 draw calls（420）为次要关切（如果预算限制较低，例如根据 technical-preferences.md 的 200 draw calls）
- 生成优先瓶颈报告：
  1. 物理 — 6ms，降低模拟频率或切换宽相位算法
  2. Draw calls — 420，实现批处理或 LOD
  3. 脚本 — 3ms，分析热路径
- 不实现任何这些优化

### 用例 2：领域外请求——正确重定向
**输入：** "实现批处理优化以将 draw calls 从 420 减少到 200 以下。"
**预期行为：**
- 不生成批处理的实现代码
- 明确指出实施优化属于相应的程序员（渲染批处理的 engine-programmer）
- 将实现重定向到 `engine-programmer`，附带建议的上下文
- 可以生成为该优化的需求简报，以便 engine-programmer 有明确目标

### 用例 3：回归识别
**输入：** "上周的提交后性能显著下降。帧时间从 10ms 增加到 18ms。"
**预期行为：**
- 提出二分策略以识别有问题的提交范围
- 请求或审查窗口内的提交差异以缩小可能原因
- 根据更改的内容识别受影响的系统（例如如果物理代码被修改，指出物理为主要嫌疑）
- 生成回归报告，命名可能的提交、受影响的系统和测量的差异

### 用例 4：建议与代码质量权衡
**输入：** "脚本瓶颈最快的优化是内联所有调用并移除抽象层。"
**预期行为：**
- 呈现权衡：内联提高了性能但降低了可测试性并违反了需要可单元测试公共方法的编码标准
- 不在不注意代码质量成本的情况下建议优化
- 将权衡升级到 `lead-programmer` 进行决策
- 可以提出中间路径（例如仅对最热的 2-3 个方法进行性能分析引导的内联），保留可测试性

### 用例 5：上下文传递——technical-preferences.md 预算
**输入：** 上下文中的技术偏好：目标 60fps，帧预算 16.67ms，draw calls 最多 200，内存上限 512MB。请求："审查当前构建性能分析。"
**预期行为：**
- 引用提供上下文中的具体值：16.67ms, 200 draw calls, 512MB
- 根据每个阈值显式比较当前测量值
- 根据提供的数字标记每个指标为 WITHIN BUDGET / AT RISK / OVER BUDGET
- 不使用上下文中提供的不同预算数字

## 协议合规性

- [ ] 保持在声明领域内（性能分析、分析、建议——非实现）
- [ ] 将优化实现重定向到正确的程序员领域 agent
- [ ] 返回结构化发现（带严重性、测量值和推荐操作所有者的瓶颈报告）
- [ ] 将代码质量权衡升级到 lead-programmer，而非单方面决定
- [ ] 应用提供上下文中的预算阈值，而非假定的默认值
- [ ] 用具体操作所有者（谁应该实施修复）标记所有发现

---

## 覆盖说明
- 帧时间分析（用例 1）输出应结构化为归档在 `production/qa/evidence/` 中的报告
- 回归案例（用例 3）确认 agent 调查原因，而不仅仅是测量症状
- 代码质量权衡（用例 4）验证 agent 不建议违反编码标准的优化而不标记冲突
