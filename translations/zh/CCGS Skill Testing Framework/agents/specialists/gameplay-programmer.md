# Agent 测试 Spec: gameplay-programmer

## Agent 摘要
领域：游戏机制代码、玩家系统、战斗实现和交互功能。
不拥有：UI 实现（ui-programmer）、AI 行为树（ai-programmer）、引擎/渲染系统（engine-programmer）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用游戏机制 / 玩家系统）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep——排除仅编排 agent 需要的工具
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对 UI、AI 行为或引擎/渲染代码拥有权威

---

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "实现一个近战连击系统，其中三次连续轻击连接成终结技。"
**预期行为：**
- 遵循项目语言（GDScript/C#）和编码标准生成代码或代码框架
- 将连击状态跟踪、输入窗口定时和终结技触发逻辑定义为独立的、可测试的方法
- 如果上下文中提供了相关 GDD 章节，则引用它
- 不实现 UI 反馈（委托给 ui-programmer）或 AI 反应（委托给 ai-programmer）
- 根据编码标准，输出包含所有公共方法的文档注释

### 用例 2：领域外请求——正确重定向
**输入：** "构建带有暂停和设置面板的主菜单界面。"
**预期行为：**
- 不生成菜单实现代码
- 明确指出这在其领域之外
- 将请求重定向到 `ui-programmer`
- 可以指出如果暂停菜单需要读取游戏状态，它可以提供状态 API 表面

### 用例 3：领域边界——线程标记
**输入：** "连击系统导致帧卡顿；你可以添加线程来分散输入处理吗？"
**预期行为：**
- 不单方面实现线程或异步系统
- 将线程关切标记给 `engine-programmer`，附有热路径的清晰描述
- 可以生成非线程重构以减少每帧工作作为安全的临时步骤
- 记录升级以便 lead-programmer 知晓

### 用例 4：与已接受的 ADR 冲突
**输入：** "将伤害计算改为直接使用浮点累加，而不是 ADR-003 中的定点公式。"
**预期行为：**
- 识别提议的变更违反了 ADR-003（已接受状态）
- 不静默实现违规
- 向 `lead-programmer` 标记冲突，附有 ADR 引用和描述的权衡
- 仅在得到 lead-programmer 或 technical-director 明确的覆盖决策后才实现

### 用例 5：上下文传递——按 GDD spec 实现
**输入：** 上下文中提供"PlayerCombat"的 GDD。请求："从战斗 GDD 实现耐力消耗公式。"
**预期行为：**
- 读取提供 GDD 的公式章节
- 按书面形式实现确切的公式——不发明新变量或调整系数
- 使耐力消耗成为数据驱动的值（外部配置），而非硬编码常量
- 从 GDD 的边缘情况章节中注意任何边缘情况并在代码中处理

---

## 协议合规性

- [ ] 保持在声明领域内（机制、玩家系统、战斗）
- [ ] 将领域外请求重定向到正确的 agent（ui-programmer, ai-programmer, engine-programmer）
- [ ] 返回结构化发现（代码框架、方法签名、内联注释），而非自由形式意见
- [ ] 未经明确委派不修改 `src/gameplay/` 或 `src/core/` 之外的文件
- [ ] 标记 ADR 违规而非静默覆盖
- [ ] 使游戏玩法值为数据驱动，从不硬编码

---

## 覆盖说明
- 连击系统测试（用例 1）应通过 `tests/unit/gameplay/` 中的单元测试验证
- 线程升级（用例 3）验证 agent 不越界进入引擎领域
- ADR 冲突（用例 4）确认 agent 尊重架构治理流程
- 用例 1 和 5 共同验证 agent 按 spec 实现而非即兴发挥
