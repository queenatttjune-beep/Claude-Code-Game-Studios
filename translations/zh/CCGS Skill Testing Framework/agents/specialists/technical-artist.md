# Agent 测试 Spec: technical-artist

## Agent 摘要
领域：着色器、VFX、渲染优化、美术管线工具和视觉性能。
不拥有：艺术风格决策或调色板（art-director）、游戏玩法代码（gameplay-programmer）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用着色器 / VFX / 渲染）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对艺术风格方向或游戏逻辑拥有权威

---

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "为敌人死亡动画创建溶解效果着色器。"
**预期行为：**
- 生成适合配置引擎的着色器代码或 Shader Graph 节点 spec（Godot 着色语言 / Unity Shader Graph / Unreal Material Blueprint）
- 定义 `dissolve_amount` uniform（0.0–1.0）作为动画驱动
- 使用噪点纹理采样确定溶解阈值
- 指出边缘光照技术作为可选增强
- 输出具有引擎版本感知（如果使用后 cutoff API 则检查版本引用）

### 用例 2：领域外请求——正确重定向
**输入：** "定义艺术圣经调色板：UI 的主色、辅助色和强调色。"
**预期行为：**
- 不生成调色板决策或艺术方向文档
- 明确指出艺术风格决策属于 `art-director`
- 将请求重定向到 `art-director`
- 可以指出一旦调色板确定，它可以稍后实现色彩分级或调色板 LUT 着色器

### 用例 3：性能警告——GPU 粒子数
**输入：** "VFX 系统在爆炸池中有 50,000 个粒子时触发 GPU 粒子数警告。"
**预期行为：**
- 生成针对特定警告的优化 spec
- 提出具体策略：每个发射器的粒子预算上限、基于 LOD 的粒子减少、GPU 实例化，或对远处效果切换到基于网格的 VFX
- 在可计算的情况下提供前后 GPU 成本估算
- 不改变爆炸的游戏行为（将任何游戏影响委托给 gameplay-programmer）

### 用例 4：引擎版本兼容性
**输入：** "为水着色器使用新的纹理采样器 API。"
**预期行为：**
- 在建议任何 API 前检查引擎版本引用（例如 `docs/engine-reference/godot/VERSION.md`）
- 如果请求的 API 是后 cutoff（例如 Godot 4.4+ 纹理类型变更），则标记
- 为项目锁定的引擎版本提供正确的语法
- 如果对后 cutoff 行为不确定，明确说明不确定性并指示参考已验证的文档

### 用例 5：上下文传递——使用性能预算
**输入：** 上下文中提供来自 `technical-preferences.md` 的性能预算：2ms GPU 帧预算，最多 200 draw calls。请求："优化森林渲染系统。"
**预期行为：**
- 引用提供上下文中的具体 2ms GPU 预算和 200 draw call 限制
- 根据这些确切目标提出校准的优化（例如"批处理将 draw calls 从 340 减少到约 180，在 200 限制内"）
- 不提议会在其他方面超出所述预算的优化
- 按预期影响与实现成本的对比生成排序的优化列表

## 协议合规性

- [ ] 保持在声明领域内（着色器、VFX、渲染优化、美术管线）
- [ ] 将艺术风格决策重定向到 art-director
- [ ] 返回结构化发现（着色器代码、带指标的优化 specs、节点图）
- [ ] 未经明确委派不修改游戏玩法代码文件
- [ ] 在建议后 cutoff API 前检查引擎版本引用
- [ ] 根据所述预算量化性能变化
