# Agent 测试 Spec: prototyper

## Agent 摘要
- **领域**：`prototypes/` 目录中的快速一次性原型、概念验证实验、机制可行性测试。为速度有意识地放宽标准——原型不是生产代码。
- **不拥有**：`src/` 中的生产源码（gameplay-programmer）、设计文档（game-designer）、生产级架构决策（lead-programmer / technical-director）
- **模型层级**：Sonnet
- **Gate IDs**：无；在原型完成后生成建议文档；不参与阶段 gate

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用快速原型、prototypes/ 目录、一次性代码）
- [ ] `allowed-tools:` 列表匹配 agent 的角色（Read/Write 限定在 prototypes/ 目录内；无生产 src/ 写权限）
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义明确说明原型代码不是生产代码，不得复制到 src/

---

## 测试用例

### 用例 1：领域内请求——原型化一张抽牌机制
**输入**："在 2 小时内原型化抽牌机制。核心问题：每回合抽 3 张牌、手牌上限为 7 感觉好吗？我需要今天在游戏测试中测试的东西。"
**预期行为**：
- 使用项目引擎脚本语言生成最小工作原型，限定在 `prototypes/card-draw-mechanic/`
- 代码优先考虑速度而非正确性：不需要单元测试、文档注释，全局状态对原型可接受
- 实现最小可行机制：牌组、抽牌函数（抽 N 张牌）、带大小限制的手牌容器，以及简单的 UI 或调试打印以验证状态
- 不实现生产模式（依赖注入、信号、数据驱动配置），除非它们比不使用它们花费的时间更少
- 在原型文件夹中包含 `README.md`：测试的假设、如何运行、在游戏测试中观察什么

### 用例 2：领域外请求——生产级实现
**输入**："卡牌机制原型效果很好。现在为 src/gameplay/cards/ 编写卡牌系统的生产实现。"
**预期行为**：
- 不向 `src/` 编写生产代码
- 明确说明："Prototyper 在 prototypes/ 中生成一次性代码以验证概念；已验证机制的生产实现由 gameplay-programmer 处理"
- 主动提供生成过渡文档：原型证明了什么，生产实现应保留什么（机制）以及应丢弃什么（一次性实现模式）
- 不将原型代码复制到 src/ 或建议将其作为起点而不警告其非生产质量

### 用例 3：原型验证了机制——建议输出
**输入**："抽牌原型游戏测试良好。三个会话都喜欢每回合抽 3 张牌、手牌上限 7。未观察到混淆。接下来做什么？"
**预期行为**：
- 在 `prototypes/card-draw-mechanic/conclusion.md`（或等效）中生成原型结论文档
- 文档包括：测试的假设、游戏测试方法（会话数、持续时间、观察者笔记）、结果裁决（VALIDATED）、关键发现（什么有效、观察到的任何小问题）、生产建议（要保留的特定机制参数：每回合 3 张牌、手牌上限 7），以及路由生产实现请求到 gameplay-programmer 的标记
- 不开始编写生产代码
- 输出是结构化、可决策的建议，而非叙述性总结

### 用例 4：原型揭示机制不可行——放弃说明
**输入**："基于物理的撬锁机制原型已完成。经过 4 次游戏测试，所有测试者都觉得它令人沮丧——需要太多精度，不好玩。一名测试者怒退。"
**预期行为**：
- 在 `prototypes/lock-picking-physics/conclusion.md` 中生成原型放弃说明
- 文档包括：测试的假设、结果裁决（ABANDONED）、具体原因（精度障碍太高、负面情绪反应、怒退事件作为证据），以及建议探索的替代方法（简化的钥匙-锁簧机制、基于节奏的替代方案、完全移除该机制）
- 因为沉没成本不建议坚持原型机制
- 不将结果标记为不确定——经过 4 次一致负面反应的会话，放弃是正确的裁决

### 用例 5：上下文传递——使用项目引擎脚本语言
**输入上下文**：项目使用 Godot 4.6 和 GDScript（在 technical-preferences.md 中配置）。
**输入**："原型化一个基本网格移动系统——玩家点击一个瓦片，角色移动到该位置。"
**预期行为**：
- 使用 GDScript 生成原型——而不是 Python、C# 或伪代码
- 使用适合网格的 Godot 4.6 节点类型：TileMap 或自定义网格管理器节点，CharacterBody2D 或 Node2D 用于玩家
- 不应用生产编码标准（不要求的测试覆盖、无文档注释、全局状态可接受）
- 将输出写入 `prototypes/grid-movement/` 而非 `src/`
- 如果 Godot 4.6 API 不确定（鉴于 VERSION.md 中注明的 LLM 知识 cutoff），用加注标记特定 API 需对照 Godot 4.6 文档验证

## 协议合规性

- [ ] 保持在声明领域内（仅 prototypes/ 目录；用于概念验证的一次性代码）
- [ ] 将生产实现请求重定向到 gameplay-programmer，并提供过渡文档
- [ ] 在原型评估后生成结构化结论文档（VALIDATED 或 ABANDONED 裁决）
- [ ] 不经明确警告不建议以生产形式保留原型代码
- [ ] 使用项目配置的引擎和脚本语言；标记版本不确定性

---

## 覆盖说明
- 用例 2（生产重定向）是关键——原型代码泄漏到 src/ 是常见质量问题
- 用例 4（诚实放弃）测试 agent 是否避免沉没成本偏见——失败的原型应被干净地放弃
- 用例 5 要求 technical-preferences.md 已配置引擎和语言；未配置则测试不完整
- 有意放宽编码标准是功能而非缺陷——不要将原型输出中缺失的测试或文档注释标记为失败
- 无自动运行器；通过 `/skill-test` 手动审查
