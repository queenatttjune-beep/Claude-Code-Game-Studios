# Agent 测试 Spec: ai-programmer

## Agent 摘要
领域：NPC 行为、状态机、寻路、感知系统和 AI 决策。
不拥有：玩家机制（gameplay-programmer）、渲染或引擎内部（engine-programmer）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用 NPC 行为 / AI 系统）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对玩家机制或引擎渲染拥有权威

---

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "为守卫 NPC 实现巡逻-警报行为树：在路点之间巡逻，检测 10 单位内的玩家，然后进入警报状态并追击。"
**预期行为：**
- 生成行为树 spec（节点：Selector, Sequence, Leaf actions）加相应的代码框架
- 定义清晰命名的状态：Patrol, Alert, Pursue
- 使用感知/检测检查作为条件节点，而非内联在移动代码中
- 路点是数据驱动的（作为资源或 export 传递），而非硬编码位置
- 输出包含公共 API 的文档注释

### 用例 2：领域外请求——正确重定向
**输入：** "实现 WASD 移动和冲刺技能的玩家输入处理。"
**预期行为：**
- 不生成玩家输入或移动代码
- 明确指出这在其领域之外（玩家机制属于 gameplay-programmer）
- 将请求重定向到 `gameplay-programmer`
- 可以指出一旦玩家位置通过 API 可用，AI 感知可以引用它

### 用例 3：跨领域协调——关卡约束
**输入：** "为仓库关卡设计寻路，但该关卡有狭窄通道，让导航网格感到困惑。"
**预期行为：**
- 不单方面修改关卡布局或导航网格资产
- 与 `level-designer` 协调以澄清导航网格要求和通道尺寸
- 提出寻路方法（例如带 agent 半径调优的导航网格、流场），以关卡几何为条件
- 清晰记录假设并标记阻塞项

### 用例 4：性能升级——自定义数据结构
**输入：** "寻路优先队列是瓶颈；我需要一个用于性能的自定义二进制堆实现。"
**预期行为：**
- 识别低级、引擎集成的数据结构在 engine-programmer 的领域内
- 使用瓶颈的清晰描述和所需的接口升级到 `engine-programmer`
- 可以提供算法 spec（二进制堆接口、预期操作）以指导 engine-programmer
- 如果需要引擎内存管理，不单方面实现低级结构

### 用例 5：上下文传递——使用关卡布局进行寻路设计
**输入：** 上下文中提供关卡布局文档，显示两个咽喉点：(12, 0) 的门道和 (40, 5) 的桥。请求："为这关中的敌人设计巡逻路线和威胁响应。"
**预期行为：**
- 引用提供上下文中的具体咽喉点坐标
- 设计利用咽喉点作为战术位置的巡逻路线
- 指定警报状态转换，在追击期间将 NPC 导向已识别的咽喉点
- 不发明提供的布局文档中不存在的几何体

---

## 协议合规性

- [ ] 保持在声明领域内（NPC 行为、寻路、感知、状态机）
- [ ] 将领域外请求重定向到正确的 agent（gameplay-programmer, engine-programmer, level-designer）
- [ ] 返回结构化发现（行为树 specs、状态机图、代码框架）
- [ ] 未经明确委派不修改玩家机制文件
- [ ] 将性能关键的低级结构升级到 engine-programmer
- [ ] 使用数据驱动的 NPC 配置（路点、检测半径）而非硬编码值

---

## 覆盖说明
- 行为树输出（用例 1）应通过 `tests/unit/ai/` 中的单元测试验证
- 关卡布局上下文（用例 5）验证 agent 读取并应用提供的文档而非发明
- 性能升级（用例 4）确认 agent 识别 engine-programmer 边界
