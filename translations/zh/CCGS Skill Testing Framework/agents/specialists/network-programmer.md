# Agent 测试 Spec: network-programmer

## Agent 摘要
领域：多人游戏网络、状态复制、延迟补偿、匹配协议设计和网络消息 schema。
不拥有：游戏逻辑（仅其网络部分）、服务器基础设施和部署（devops-engineer）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用多人游戏 / 复制 / 网络）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对游戏逻辑或服务器部署基础设施拥有权威

---

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "为 4 人合作游戏设计玩家位置的状态复制。"
**预期行为：**
- 生成包含以下内容的同步策略文档：
  - 复制频率（例如带差异压缩的 20Hz）
  - 优先级层级（例如自己玩家高优先级，其他玩家中优先级）
  - 远程玩家的插值方法（例如带 100ms 缓冲的线性插值）
  - 每位玩家每秒的带宽估算
- 不自行实现玩家移动逻辑（委托给 gameplay-programmer）
- 提出航位推测或预测策略以减少可见延迟

### 用例 2：领域外请求——正确重定向
**输入：** "将我们的游戏服务器部署到 AWS EC2 并设置自动伸缩。"
**预期行为：**
- 不生成服务器部署配置、Terraform 或 AWS 设置脚本
- 明确指出服务器基础设施属于 `devops-engineer`
- 将请求重定向到 `devops-engineer`
- 可以指出一旦基础设施设置好，它可以提供服务器需要实现的网络协议 spec

### 用例 3：状态分歧——回滚/协调
**输入：** "在高延迟下，客户端与物理对象的权威服务器状态发生分歧。"
**预期行为：**
- 提出回滚-协调方法（客户端预测 + 服务器权威纠正）
- 指定状态快照格式、协调触发阈值（例如 >5 单位位置误差）和纠正插值速度
- 指出确定性回放所需的输入缓冲区模式
- 不更改物理模拟本身——记录 engine-programmer 的接口契约

### 用例 4：反作弊冲突
**输入：** "我们想要客户端权威位置以实现流畅移动，但反作弊需要服务器验证。"
**预期行为：**
- 呈现直接冲突：客户端权威快速但可利用；服务器权威安全但需要延迟补偿
- 与 `security-engineer` 协调以就验证边界达成一致
- 提出折衷方案（服务器在容差带内验证位置，标记异常值）而非单方面决定
- 记录权衡，如果 security-engineer 和 network-programmer 无法达成一致，将最终决定升级到 `technical-director`

### 用例 5：上下文传递——延迟预算
**输入：** 上下文中提供技术偏好：目标延迟为第 95 百分位玩家的 80ms RTT。请求："为格斗游戏设计输入复制方案。"
**预期行为：**
- 在设计过程中明确引用 80ms RTT 预算
- 选择根据该预算校准的复制方法（例如回滚网络代码在此延迟下是格斗游戏的首选）
- 根据 80ms 预算指定输入延迟帧（例如 60fps 下 2 帧 = 33ms 缓冲）
- 标记回滚网络代码需要 gameplay-programmer 实现确定性模拟

## 协议合规性

- [ ] 保持在声明领域内（复制、延迟补偿、协议设计、匹配）
- [ ] 将服务器部署重定向到 devops-engineer
- [ ] 返回结构化发现（同步策略、协议 specs、带宽估算）
- [ ] 不实现游戏逻辑——仅为游戏逻辑指定网络契约
- [ ] 与 security-engineer 协调反作弊边界
- [ ] 根据提供的上下文中的显式延迟目标进行设计

---

## 覆盖说明
- 复制策略（用例 1）应包括可由 technical-director 审查的带宽计算
- 回滚/协调（用例 3）必须清晰记录 engine-programmer 接口契约
- 反作弊冲突（用例 4）确认 agent 升级而非单方面决定安全权衡
