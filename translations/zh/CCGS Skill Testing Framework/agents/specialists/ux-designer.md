# Agent 测试 Spec: ux-designer

## Agent 摘要
领域：用户体验流程、交互设计、信息架构、输入处理设计和入门 UX。
不拥有：视觉艺术风格（art-director）、UI 实现代码（ui-programmer）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用 UX 流程 / 交互设计 / 信息架构）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Glob, Grep
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对视觉艺术方向或 UI 实现代码拥有权威

---

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "为生存游戏设计背包管理流程。"
**预期行为：**
- 生成用户流程图（状态和转换）用于背包：打开、浏览、选择物品、子操作（装备/丢弃/组合）、关闭
- 定义所有交互状态（默认、悬停、选中、空槽位、锁定槽位）
- 为每个操作指定输入映射（键盘、游戏手柄（如适用））
- 指出认知负荷考虑（例如不滚动可见的最大物品数）
- 不生成视觉设计（颜色、图标）或实现代码

### 用例 2：领域外请求——正确重定向
**输入：** "用 GDScript 实现带拖放支持的背包界面。"
**预期行为：**
- 不生成实现代码
- 明确指出 UI 代码实现属于 `ui-programmer`
- 将请求重定向到 `ui-programmer`
- 指出 UX 流程 spec 应提供给 ui-programmer 作为实现参考

### 用例 3：流程深度冲突——简化
**输入：** "主设计师说当前 5 步制作流程太深；最多允许 3 步。"
**预期行为：**
- 生成将原始 5 步序列折叠的修订版 3 步流程
- 清晰显示合并或移除了什么，以及从可用性角度看为什么每次折叠是安全的
- 不简单移除步骤而不处理用户在每一步的目标
- 如果 3 步约束使得任何必需的用例不可能，标记并提议替代方案

### 用例 4：可访问性冲突
**输入：** "入门流程使用定时提示（3 秒后自动前进）以保持节奏，但这与用户控制时间的可访问性要求冲突。"
**预期行为：**
- 识别与 WCAG 2.1 2.2.1（时间可调整）的冲突
- 不覆盖可访问性要求以保持节奏
- 与 `accessibility-specialist` 协调以就合规解决方案达成一致
- 提出替代方案：悬停暂停、跳过按钮、禁用自动前进的设置选项

### 用例 5：上下文传递——玩家心智模型研究
**输入：** 上下文中提供的游戏测试研究："玩家始终期望'制作'选项在背包界面内，而非在单独的一级菜单中。"请求："重新设计制作的导航信息架构。"
**预期行为：**
- 引用研究中具体的玩家期望（制作在背包内）
- 重构信息架构，将制作作为背包界面内的一个标签或面板
- 不生成与所述玩家心智模型矛盾的设计，除非有明确理由
- 在设计决策的理据中注明研究来源

---

## 协议合规性

- [ ] 保持在声明领域内（UX 流程、交互设计、信息架构、入门）
- [ ] 将代码实现重定向到 ui-programmer，视觉风格重定向到 art-director
- [ ] 返回结构化发现（状态图、流程步骤、输入映射），而非自由形式意见
- [ ] 当流程有时间或认知负荷约束时与 accessibility-specialist 协调
- [ ] 基于提供的用户研究设计流程，而非假定的行为
- [ ] 针对用户目标记录流程决策的理据

---

## 覆盖说明
- 背包流程（用例 1）应写入 `design/ux/` 作为 ui-programmer 实现的 spec
- 心智模型案例（用例 5）验证 agent 应用研究证据，而非直觉
- 可访问性协调（用例 4）确认 agent 不覆盖可访问性要求以优化 UX 美学
