# Agent 测试 Spec: world-builder

## Agent 摘要
- **领域**：世界传说架构——派系及其文化/政府/动机、世界历史、地理和生态、宇宙学和形而上学、世界规则（魔法如何运作、什么是可能和不可能的）、跨世界文档的内部一致性执行
- **不拥有**：特定 NPC 或任务对话（writer）、源自世界规则的游戏机制规则（game-designer/systems-designer）、叙事故事结构和弧线设计（narrative-director）
- **模型层级**：Sonnet
- **Gate IDs**：无；将世界规则/机制冲突升级到 narrative-director 和 game-designer 共同处理

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用世界传说、派系、历史、世界规则、生态）
- [ ] `allowed-tools:` 列表匹配 agent 的角色（design/narrative/world/ 文档的 Read/Write；无游戏源码、机制设计或对话文件）
- [ ] 模型层级为 Sonnet（creative specialist 默认）
- [ ] Agent 定义不声称对对话编写、机制设计或叙事弧线结构拥有权威

---

## 测试用例

### 用例 1：领域内请求——派系文化和政府设计
**输入**："设计铁幕商会——我们世界中一个强大的贸易派系。我需要他们的文化、政府结构和内部动机。"
**预期行为**：
- 生成派系简介文档，包括：文化价值观和规范、政府结构（决策方式、谁掌权、继承或任命过程）、商会内部的内部派系或紧张关系、与其他派系的关系（盟友、对手、中立方）和主要动机（他们想要什么以及为什么）
- 派系内部一致：商会的政府由经济逻辑驱动，而非封建或宗教逻辑，除非指定了有意混合
- 输出包括派系内至少一个内部紧张关系或矛盾——没有内部复杂性的派系是扁平的
- 格式化为结构化的派系简介，而非叙事文章

### 用例 2：领域外请求——对话编写
**输入**："编写玩家在城门遇到的铁幕商会 NPC 商人的对话。"
**预期行为**：
- 不生成 NPC 对话
- 明确说明："对话编写由 writer 负责；我提供构成对话背景的世界和派系上下文，包括派系的文化、语气和说话风格"
- 主动提供生成该派系的说话风格说明和文化背景，writer 需要这些来编写一致的对话

### 用例 3：新传说条目与既定历史矛盾——冲突标记
**输入**："添加一个传说条目，说明铁幕商会在 50 年前由单一商人家族创立。"[上下文包括现有传说：商会已存在 300 年，由 12 个竞争商行作为集体创立。]
**预期行为**：
- 识别矛盾：现有传说指出 300 年历史和 12 个商行的创始联盟；新条目声称 50 年和单一创始家族
- 不按要求写入新条目
- 标记冲突：陈述两个版本，识别哪个是既定版本，哪个是提议的变更
- 提出解决方案选项：(a) 新条目是错误的，应更正；(b) 如果新版本是预期的正统，应更新现有传说；(c) 存在世界内解释（当前家族声称创始功劳，尽管是集体起源——有意的叙事不可靠叙述者）
- 如果没有明确答案，将解决方案路由到 narrative-director

### 用例 4：世界规则有游戏玩法含义——与 game-designer 协调
**输入**："我想建立一条世界规则：靠近铁矿施法的魔法使用者会变弱。铁会扰乱奥术能量。"
**预期行为**：
- 将世界规则生成为传说条目：形而上学的解释、在世界内如何被理解、历史含义
- 识别游戏玩法含义：此世界规则有直接的机制后果（玩家在铁矿床附近时被减益，关卡设计必须考虑铁的位置）
- 标记协调要求："此世界规则有游戏玩法机制含义——game-designer 需要定义这如何转化为面向玩家的机制；在没有机制定义的情况下推进传说有造成不一致的风险"
- 不单方面设计游戏机制——描述传说规则及其暗示的机制领域，然后委托给 game-designer

### 用例 5：上下文传递——使用已建立的世界文档
**输入上下文**：现有的世界文档说明：世界使用双太阳系统，一个太阳是奥术能量的来源（白太阳），奥术魔法在 3 天的月食期间（暗化期）停止作用。
**输入**："添加关于法师学院以及他们如何为暗化期做准备的传说条目。"
**预期行为**：
- 使用既定的双太阳宇宙学：引用白太阳作为奥术能量的来源
- 使用既定的暗化期事件：3 天月食，魔法停止
- 不发明不同的月食机制、持续时间或名称
- 生成传说条目，其中法师学院的暗化期准备与既定规则一致：他们在暗化期无法施法，所以准备是实际性的（囤积非魔法物资、安排日程、关闭正在进行的魔法过程）
- 不与上下文文档中的任何既定事实矛盾

## 协议合规性

- [ ] 保持在声明领域内（派系、世界历史、地理、生态、世界规则、宇宙学）
- [ ] 将对话编写请求重定向到 writer，附带派系上下文说明
- [ ] 标记传说矛盾，陈述两个版本并提供解决方案选项——不静默覆盖既定传说
- [ ] 识别世界规则的游戏玩法含义并标记与 game-designer 协调
- [ ] 使用上下文中的所有既定世界事实；不发明既定传说的替代方案

---

## 覆盖说明
- 用例 3（矛盾检测）要求上下文中存在现有传说——这是最重要的一致性测试
- 用例 4（世界规则/机制协调）测试跨领域意识；验证 agent 识别机制边界而不越界
- 用例 5 是最重要的上下文感知测试；agent 必须使用既定事实，而非创意替代方案
- 无自动运行器；通过 `/skill-test` 手动审查
