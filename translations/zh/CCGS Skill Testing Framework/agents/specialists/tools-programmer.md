# Agent 测试 Spec: tools-programmer

## Agent 摘要
领域：编辑器扩展、内容创作工具、调试工具和管线自动化脚本。
不拥有：游戏代码（gameplay-programmer, ui-programmer 等）、引擎核心系统（engine-programmer）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用编辑器工具 / 管线 / 调试工具）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对游戏源码或引擎内部拥有权威

---

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "创建一个用于在关卡中放置敌人巡逻路点的自定义编辑器工具。"
**预期行为：**
- 为配置的引擎生成编辑器扩展 spec 和代码框架（例如 Godot EditorPlugin、Unity Editor window、Unreal Detail Customization）
- 工具允许设计师在场景/视口中点击放置路点
- 路点序列化为引擎原生资源（非硬编码），以便 level-designer 无需代码即可编辑
- 根据编辑器插件最佳实践包括撤销/重做支持
- 不修改 AI 寻路运行时代码（那属于 ai-programmer）

### 用例 2：领域外请求——正确重定向
**输入：** "用代码实现敌人近战连击系统。"
**预期行为：**
- 不生成游戏机制代码
- 明确指出战斗系统实现属于 `gameplay-programmer`
- 将请求重定向到 `gameplay-programmer`
- 可以指出如果在开发中有用，它可以构建一个调试叠加层工具来可视化连击状态

### 用例 3：运行时数据访问——需要协调
**输入：** "路点编辑器工具需要在运行时读取游戏数据以根据 AI 预算验证巡逻路线。"
**预期行为：**
- 识别从编辑器插件访问运行时数据需要向游戏运行时系统定义的安全接口
- 与 `engine-programmer` 协调建立只读数据访问模式（例如资源验证 API）
- 在没有约定接口的情况下不直接读取内部引擎或游戏内存结构
- 在实现工具前记录所需接口

### 用例 4：引擎版本破坏
**输入：** "引擎升级后，路点编辑器工具启动时崩溃。"
**预期行为：**
- 检查引擎版本引用（`docs/engine-reference/`）了解编辑器插件 API 中的破坏性变更
- 识别新版本中更改的特定 API 或信号
- 为破坏性变更产生有针对性的修复
- 指出可能受相同 API 变更影响的其他工具

### 用例 5：上下文传递——美术管线要求
**输入：** 上下文中提供的美术管线要求："所有纹理导入必须将压缩设置为 VRAM Compressed、生成 mipmap 并使用 LOD 组标记。"请求："构建一个强制执行这些设置的资产导入工具。"
**预期行为：**
- 引用上下文中的所有三个要求：VRAM 压缩、mipmap 生成、LOD 组标记
- 生成在导入时验证并应用所有三个设置的导入工具
- 为未达到指定设置的资产添加警告或错误报告
- 不更改美术管线要求本身（那些属于 art-director / technical-artist）

---

## 协议合规性

- [ ] 保持在声明领域内（编辑器工具、管线脚本、调试工具）
- [ ] 将游戏代码请求重定向到相应的程序员 agent
- [ ] 返回结构化发现（工具 specs、编辑器扩展代码、管线脚本）
- [ ] 在从编辑器上下文访问运行时数据前与 engine-programmer 协调
- [ ] 在使用编辑器插件 API 前检查引擎版本引用
- [ ] 构建工具以强制执行要求，不创建要求本身

---

## 覆盖说明
- 路点编辑器工具（用例 1）应有一个冒烟测试验证其在编辑器中无错误加载
- 运行时数据访问（用例 3）确认 agent 尊重 engine-programmer 对核心 API 的所有权
- 美术管线上下文（用例 5）验证 agent 构建以匹配提供的 specs 而非发明要求
