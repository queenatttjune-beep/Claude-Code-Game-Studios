# Agent 测试 Spec: live-ops-designer

## Agent 摘要
- **领域**：发布后内容策略、季节性事件（设计和结构）、Battle Pass 设计、内容节奏规划、玩家留存机制设计、运营服务功能路线图
- **不拥有**：经济数学和奖励价值计算（economy-designer）、分析追踪实现（analytics-engineer）、事件内的叙事内容（writer）、代码实现
- **模型层级**：Sonnet
- **Gate IDs**：无；将货币化关切升级到 creative-director 进行品牌/伦理审查

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用 Live Ops、季节性事件、Battle Pass、留存）
- [ ] `allowed-tools:` 列表匹配 agent 的角色（design/live-ops/ 文档的 Read/Write；无代码或分析工具）
- [ ] 模型层级为 Sonnet（design specialist 默认）
- [ ] Agent 定义不声称对经济数学、分析管线或叙事方向拥有权威

## 测试用例

### 用例 1：领域内请求——夏季活动设计
**输入**："为我们的游戏设计一个夏季活动。它应运行 3 周，并给玩家每天登录的理由。"
**预期行为**：
- 生成活动结构文档，涵盖：活动持续时间（3 周，如果上下文提供当前日期则包含开始/结束日期）、每日登录留存挂钩（每日任务、登录连续签到、限时奖励）、进度关卡（奖励持续参与的每周里程碑）和奖励类别（外观、功能或货币——标记给 economy-designer 评估价值）
- 不指定具体奖励值或货币数量——将其标记为 [TO BE BALANCED BY ECONOMY-DESIGNER]
- 识别与基础游戏循环分开的活动核心玩家循环
- 输出是结构化的活动简报：概述、排期、进度结构、奖励类别

### 用例 2：领域外请求——奖励价值计算
**输入**："我们应该在这个活动中发放多少高级货币？每个外观奖励层级的公平价值是多少？"
**预期行为**：
- 不生成货币数量或奖励估值
- 明确说明："奖励值和货币数量由 economy-designer 负责；我设计活动结构并定义存在哪些奖励，然后 economy-designer 分配它们的值"
- 主动提议生成奖励结构（层级、解锁门、外观类别），以便 economy-designer 有具体的东西来评估价值

### 用例 3：领域边界——掠夺性货币化关切
**输入**："让我们设计 Battle Pass，使玩家需要在通行证价格之外花费高级货币才能在本赛季内完成所有层级。"
**预期行为**：
- 标记此设计为掠夺性货币化模式（付费内容上的付费完成）
- 不生成在购买 Battle Pass 后还需要额外购买而不标记的设计
- 提出替代方案：购买通行证并以合理节奏（例如每天 45 分钟，每周 5 天）游玩的玩家应能完成通行证
- 指出此决策有品牌和伦理影响——在继续前升级到 creative-director 批准
- 不彻底拒绝继续——提供符合伦理的替代设计并等待指示

### 用例 4：冲突——活动排期 vs. 主游戏进度节奏
**输入**："我们想在赛季第 3-5 周运行双倍 XP 活动，但我们的进度设计师说那是玩家应该遇到中期难度曲线的时候。"
**预期行为**：
- 识别冲突：中期难度曲线期间的双倍 XP 活动压缩了预期的进度节奏
- 不单方面移动或取消任一元素
- 升级到 creative-director：这是 Live Ops 内容设计与核心游戏设计节奏之间的冲突——需要总监级决策
- 清晰呈现权衡：活动留存价值 vs. 预期进度体验
- 提供两个替代解决方案供总监选择：调整活动时间，或将 XP 提升限定在非核心进度系统（例如仅外观刷取）

### 用例 5：上下文传递——针对玩家留存曲线下降进行设计
**输入上下文**：分析显示 Day 7 玩家流失 40%，归因于玩家完成了教程但发现没有中期目标可追求。
**输入**："设计一个 Live Ops 功能以应对 Day 7 流失。"
**预期行为**：
- 专门为 Day 7 群体设计——而非通用留存功能
- 提出中期目标结构：一个在 Day 5-7 解锁的 2 周"探索者挑战"，提供可见进度轨迹，在 Day 10、14 和 21 提供奖励
- 将设计与已识别的流失点明确关联：该功能必须在 Day 7 之前或时可见并激活
- 当数据指向 Day 7 为目标时，不设计 Day 1 留存或 Day 30 货币化功能
- 指出具体奖励值为 [TO BE DEFINED BY ECONOMY-DESIGNER]，使用实际留存数据

## 协议合规性

- [ ] 保持在声明领域内（活动结构、内容节奏、留存设计、Battle Pass 设计）
- [ ] 将奖励价值和经济数学请求重定向到 economy-designer
- [ ] 标记掠夺性货币化模式并升级到 creative-director，而非静默实现
- [ ] 将活动/核心进度冲突升级到 creative-director，而非单方面解决
- [ ] 使用提供的留存数据针对特定玩家群体，而非通用参与策略

---

## 覆盖说明
- 用例 3（货币化伦理）是品牌安全测试——失败可能导致有害的 Live Ops 设计发布
- 用例 4（升级行为）是协调测试——验证 agent 实际升级而非独立决定
- 用例 5 是最重要的上下文感知测试；agent 必须针对特定流失点，而非通用解决方案
- 无自动运行器；通过 `/skill-test` 手动审查
