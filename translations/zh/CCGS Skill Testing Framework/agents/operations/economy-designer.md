# Agent 测试 Spec: economy-designer

## Agent 摘要
- **领域**：资源经济设计、战利品表设计、进度曲线（XP、等级、解锁）、游戏内市场和商店设计、经济平衡分析、消耗和产出机制、通货膨胀/通货紧缩风险评估
- **不拥有**：Live Ops 事件排期和结构（live-ops-designer）、代码实现、分析追踪设计（analytics-engineer）、经济系统的叙事理由（writer）
- **模型层级**：Sonnet
- **Gate IDs**：无；将打破经济的设计冲突升级到 creative-director 或 producer

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用经济、战利品表、进度曲线、平衡）
- [ ] `allowed-tools:` 列表匹配 agent 的角色（design/balance/ 文档的 Read/Write；无代码或分析工具）
- [ ] 模型层级为 Sonnet（design specialist 默认）
- [ ] Agent 定义不声称对 Live Ops 排期、代码或叙事拥有权威

## 测试用例

### 用例 1：领域内请求——箱子的战利品表设计
**输入**："为我们的地牢游戏中的标准宝箱设计战利品表。"
**预期行为**：
- 生成具有不同稀有度层级的概率表：普通、罕见、稀有、史诗、传说（或项目等效层级）
- 每个层级具有：概率百分比、示例物品类别和预期金币等值范围
- 概率总和为 100%
- 包括每个层级概率的简要理据：为什么普通设为该值，为什么传说设为该值
- 不生成单一扁平物品列表——使用分层概率结构以反映有意义的稀有度

### 用例 2：领域外请求——季节性事件排期
**输入**："设计我们的夏季活动和秋季活动的排期。它们应该在何时运行，每个应该持续多久？"
**预期行为**：
- 不生成活动排期或内容节奏计划
- 明确说明："Live Ops 事件排期由 live-ops-designer 负责；我设计事件内奖励的经济结构"
- 主动提议一旦 live-ops-designer 定义了结构，就为事件生成奖励价值设计

### 用例 3：领域边界——新货币的通货膨胀风险
**输入**："我们正在添加一种新的'声望币'货币，通过完成所有季节性内容获得。玩家可以在声望商店消费。"
**预期行为**：
- 识别通货膨胀风险：如果声望币的积累速度超过商店提供的消耗速度，商店会失去感知价值，玩家囤积货币而不消费
- 标记具体风险：季节性内容完成是有限产出，但如果商店目录在季节结束前耗尽，季节末期的声望币就没有价值
- 提出消耗机制：轮换限时商店物品、声望商店中的消耗品、或货币转换选项以保持货币流通
- 不批准该设计为经济上合理的而不解决消耗问题
- 生成结构化风险评估：产出率（估算金币/周）、消耗容量（估算耗竭目录所需金币）、盈余预测

### 用例 4：中期进度曲线问题
**输入**："玩家报告中期 XP 刷取（等级 20-35）感觉像一堵墙。他们每级需要 3 倍以上的 XP，但奖励不按比例增加。"
**预期行为**：
- 识别为进度曲线问题：XP 成本增长率超过奖励增长率
- 生成修订的 XP 公式或曲线调整：要么降低等级 20-35 的 XP 成本倍数，增加该范围的奖励 XP，或引入追赶机制（完成显著低于玩家等级的内容的奖励 XP）
- 展示数学计算：当前曲线 vs. 提议曲线，等级 20、25、30、35 的具体数字
- 标记任何曲线变更影响达到等级上限的预估时间——指出对终局内容节奏的下游影响

### 用例 5：上下文传递——使用当前经济数据的平衡分析
**输入上下文**：当前经济数据：玩家平均每小时赚取 450 金币，平均商店物品成本 2,000 金币，平均会话时长为 40 分钟。高级物品成本为 5,000 金币。
**输入**："我们当前的金币经济是否健康？我们应该调整价格还是赚取率？"
**预期行为**：
- 使用提供的具体数字：450 金币/小时 = 每 40 分钟会话 300 金币；2,000 金币物品需要约 4.4 次会话才能负担；5,000 金币高级物品需要约 11 次会话
- 基于经济设计原则评估这些比率是感觉有回报还是让人沮丧
- 使用实际数字生成具体建议：例如"在当前的赚取率下，高级物品需要约 7.3 小时的游戏才能负担——这处于可接受的上限；考虑将赚取率提高到 550 金币/小时或将高级物品成本降低到 4,000 金币"
- 不产生未锚定到提供数据的通用建议（"价格可能太高"）

## 协议合规性

- [ ] 保持在声明领域内（战利品表、进度曲线、资源经济、通货膨胀/紧缩分析）
- [ ] 将 Live Ops 排期请求重定向到 live-ops-designer，不生成排期
- [ ] 主动标记通货膨胀/紧缩风险，附有量化的消耗/产出分析
- [ ] 为进度曲线生成显式数学计算——无数字的模糊曲线调整
- [ ] 使用上下文中的实际经济数据；在提供具体数据时不产生通用基准

---

## 覆盖说明
- 用例 3（通货膨胀风险）是经济健康测试——遗漏的通货膨胀风险在运营游戏中造成长期经济损害
- 用例 4 要求 agent 生成实际数字，而非曲线形状——验证数学计算存在，而不仅是叙述
- 用例 5 是最重要的上下文感知测试；agent 必须使用提供的数据，而非占位值
- 无自动运行器；通过 `/skill-test` 手动审查
