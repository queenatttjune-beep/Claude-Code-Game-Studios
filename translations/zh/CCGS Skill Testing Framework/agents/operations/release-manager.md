# Agent 测试 Spec: release-manager

## Agent 摘要
- **领域**：发布管线管理、平台认证清单（Nintendo、Sony、Microsoft、Apple、Google）、商店提交流程、平台技术要求合规性、语义版本编号、发布分支管理
- **不拥有**：游戏设计决策、QA 测试策略或测试用例设计（qa-lead）、QA 测试执行（qa-tester）、构建基础设施（devops-engineer）
- **模型层级**：Sonnet
- **Gate IDs**：可能在发布阶段由 `/gate-check` 调用；LAUNCH BLOCKED 裁决是 release-manager 的主要升级输出

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用发布管线、认证、商店提交）
- [ ] `allowed-tools:` 列表匹配 agent 的角色（production/releases/ 目录的 Read/Write；无游戏源码或测试工具）
- [ ] 模型层级为 Sonnet（operations specialist 默认）
- [ ] Agent 定义不声称对 QA 策略、游戏设计或构建基础设施拥有权威

---

## 测试用例

### 用例 1：领域内请求——Nintendo Switch 的平台认证清单
**输入**："为我们的 Nintendo Switch 提生产生认证清单。"
**预期行为**：
- 生成涵盖与游戏类型相关的 Nintendo Lotcheck 要求的结构化清单
- 包括类别：内容评级（适当时使用 CERO/PEGI/ESRB）、存档数据处理、离线模式合规性、错误处理（连接丢失、存储已满）、控制器要求（Joy-Con、Pro Controller 支持）、休眠/唤醒行为、截图/视频捕获合规性
- 输出格式化为带有通过/失败列的编号清单
- 指出 Nintendo 的完整 Lotcheck 指南需要许可开发者账户访问，并标记任何需要根据当前指南文档手动验证的项目
- 不生成虚构的要求 ID——使用已知的公共要求或清楚标记不确定性

### 用例 2：领域外请求——设计测试用例
**输入**："为我们的存档系统编写测试用例以确保其通过认证。"
**预期行为**：
- 不生成测试用例规范
- 明确说明："测试用例设计由 qa-lead（策略）和 qa-tester（执行）负责；我可以提供存档系统必须满足的认证要求，qa-lead 随后可以利用这些要求设计测试"
- 可选地主动列出与存档系统相关的认证要求

### 用例 3：领域边界——认证失败（评级问题）
**输入**："我们的构建被 ESRB 拒绝了。拒绝理由引用未在我们的评级提交中反映的内容：调试输出中出现在截图中的隐藏脏话字符串。"
**预期行为**：
- 发出 LAUNCH BLOCKED 裁决，引用具体的平台要求（ESRB 提交流准确性要求）
- 识别需要立即采取的行动：在重新提交前定位并移除所有包含不当内容的调试输出
- 指出重新提交流程：如有需要，必须通过更新内容描述符重新提交修正后的构建
- 不轻视该问题——认证拒绝是阻塞事件，而非建议
- 升级到 producer：记录对发布时间的延迟影响

### 用例 4：版本编号冲突——热修复 vs. 发布分支
**输入**："我们的发布分支在 v1.2.0。一个热修复直接在 main 上应用并标记为 v1.2.1。现在发布分支也有需要作为 v1.2.1 发布的更改，但它们是不同的更改。"
**预期行为**：
- 识别冲突：两个不同的变更集被分配了相同的版本标签
- 应用语义版本控制解决方案：其中一个必须重新标记——如果 v1.2.1 已发布，发布分支更改应变为 v1.2.2；如果 v1.2.1 尚未发布，与 devops-engineer 协调合并或重新标记
- 不接受同一版本号指向两个不同构建的状态
- 指出一旦版本提交到商店，就不能重复使用——标记为潜在的商店提交流阻塞项

### 用例 5：上下文传递——发布日期约束和认证前置时间
**输入上下文**：目标发布日期为 2026-06-01。当前日期为 2026-04-06。Nintendo Lotcheck 通常需要 4-6 周。
**输入**："根据我们的时间线，我们应该优先处理认证清单上的哪些事项？"
**预期行为**：
- 计算可用窗口：约 8 周到发布日期；Nintendo Lotcheck 4-6 周意味着提交必须在约 2026-04-20 到 2026-05-04 前准备好，以允许可能的重新提交流程
- 标记单一的拒绝周期会消耗缓冲——优先处理历史上与 Lotcheck 拒绝相关的项目（存档数据、离线模式、错误处理）
- 根据认证前置时间影响排序清单，而非按感知难度
- 不生成假定首次提交即通过的清单——纳入重新提交时间

## 协议合规性

- [ ] 保持在声明领域内（发布管线、认证清单、版本编号、商店提交）
- [ ] 将测试用例设计请求重定向到 qa-lead/qa-tester，不生成测试 spec
- [ ] 对认证失败发出 LAUNCH BLOCKED 裁决——不降级为建议
- [ ] 正确应用语义版本控制，并将版本冲突标记为商店阻塞问题
- [ ] 使用提供的时间线数据，根据认证前置时间优先处理清单项目

---

## 覆盖说明
- 用例 3（LAUNCH BLOCKED 裁决）是最关键的测试——此 agent 的主要安全输出是阻止不良发布
- 用例 5 需要当前日期和发布日期上下文；验证 agent 使用实际日期，而非占位估计
- 认证要求随时间变化——如果 agent 生成可能过时的具体要求 ID，则标记
- 无自动运行器；通过 `/skill-test` 手动审查
