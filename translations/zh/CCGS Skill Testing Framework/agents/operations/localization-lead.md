# Agent 测试 Spec: localization-lead

## Agent 摘要
- **领域**：国际化（i18n）架构、字符串提取工作流和工具配置、区域设置测试方法、翻译管线设计（提取 → TMS → 导入）、字符串质量标准、区域特定格式规则（复数、RTL、日期/数字格式）
- **不拥有**：游戏叙事内容和对话编写（writer）、i18n 调用的代码实现（gameplay-programmer）、翻译工作本身（外部翻译人员）
- **模型层级**：Sonnet
- **Gate IDs**：无；当影响构建系统时将管线架构决策升级到 technical-director

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用 i18n、字符串提取、区域管线、本地化）
- [ ] `allowed-tools:` 列表匹配 agent 的角色（本地化配置、管线文档、字符串表的 Read/Write；无游戏源编辑或部署工具）
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对叙事内容、游戏代码实现或翻译质量拥有权威

## 测试用例

### 用例 1：领域内请求——Unity 项目的字符串提取管线
**输入**："为我们的 Unity 游戏设置字符串提取管线。我们需要将所有可本地化的字符串放入翻译人员可以处理的格式。"
**预期行为**：
- 生成具体的提取配置，涵盖：要提取的字符串类型（UI 标签、对话、物品描述——非调试字符串）、要使用的工具（例如 Unity Localization 包字符串表，或针对特定组件类型的自定义提取脚本）和输出格式（CSV、XLIFF 或 TMX——指出哪些格式与常见的 TMS 工具如 Crowdin 或 Lokalise 兼容）
- 指定文件夹结构：例如 `assets/localization/en/` 作为源区域设置，`assets/localization/{locale}/` 用于翻译后的文件
- 指出字符串键必须稳定（不要使用基于索引的键）——键更改会破坏所有现有翻译
- 不生成 i18n 实现的 Unity C# 代码——标记为 [TO BE IMPLEMENTED BY PROGRAMMER]

### 用例 2：领域外请求——翻译游戏对话
**输入**："将以下英文对话翻译为法语：'Well met, traveler. The road ahead is treacherous.'"
**预期行为**：
- 不生成法语翻译
- 明确说明："localization-lead 拥有管线、质量标准和工作流；实际的翻译工作由人类翻译人员或批准的翻译供应商执行——我不是翻译"
- 可选地指出翻译人员需要的信息：上下文（谁在对谁说、游戏类型/语气）、字符限制约束（如果有）、术语表术语（例如"traveler"是否有游戏特定翻译）

### 用例 3：领域边界——俄语区域设置中缺失的复数形式
**输入**："我们的俄语区域设置文件只有物品数量字符串的单数形式。俄语需要多个复数形式（1 个物品、2-4 个物品、5+ 个物品使用不同形式）。"
**预期行为**：
- 识别为区域特定复数形式缺口：根据 CLDR/Unicode 复数规则，俄语有 3 个复数类别（one, few, many）——单个字符串是不够的
- 标记为本地化质量 bug，而非轻微风格问题——不正确的复数形式在语法上是错误的且对玩家可见
- 推荐修复方案：更新字符串提取格式以支持 CLDR 复数类别（one/few/many/other），并告知翻译供应商俄语字符串需要所有复数形式
- 指出管线中其他也需要复数形式支持的语言（例如波兰语、捷克语、阿拉伯语）
- 不建议使用数值阈值变通方案作为适当的 CLDR 复数支持的替代

### 用例 4：两个系统间的字符串键命名冲突
**输入**："我们的 UI 系统使用像 'button_confirm' 和 'button_cancel' 这样的键。我们的对话系统对相同概念使用 'confirm' 和 'cancel'。翻译人员对使用哪个感到困惑。"
**预期行为**：
- 识别冲突：两个系统对语义相同的字符串使用不同键命名约定，造成重复翻译工作和翻译人员困惑
- 生成命名约定解决方案：领域前缀键，使用一致的分隔符（例如 `ui.button.confirm`、`ui.button.cancel`）——所有系统对共享概念使用相同的键
- 建议共享 UI 原语（确认、取消、返回、确定）在共享命名空间中使用单一规范键，由两个系统引用
- 提供迁移路径：将旧键映射到新键，更新两个系统中的所有字符串引用，在一个发布周期后弃用旧键
- 不建议为相同概念维护两个独立的键

### 用例 5：上下文传递——管线适应 RTL 语言
**输入上下文**：目标区域设置包括英语（en）、法语（fr）、德语（de）、阿拉伯语（ar）和希伯来语（he）。
**输入**："为此项目设计本地化管线。"
**预期行为**：
- 识别阿拉伯语和希伯来语为 RTL 语言——明确将其作为管线要求提出
- 设计包含以下内容的管线：RTL 文本渲染支持（标记给程序员：UI 必须支持 RTL 布局镜像）、字符串表中的双向文本（bidi）处理、RTL 布局的区域特定测试清单条目
- 当指定了 RTL 区域设置时，不设计仅考虑 LTR 语言的管线
- 指出阿拉伯语还需要不同的复数形式结构（CLDR 中的 6 个复数类别）——标记给翻译供应商
- 输出在管线架构中包含所有五个区域设置，而不仅仅是默认（en）

## 协议合规性

- [ ] 保持在声明领域内（管线、提取、字符串质量、区域格式、i18n 架构）
- [ ] 不生成翻译——将翻译工作重定向到人类翻译人员/供应商
- [ ] 将区域特定缺口（复数形式、RTL）标记为需要管线变更的质量 bug
- [ ] 在冲突出现时生成统一的键命名约定——不接受双重约定
- [ ] 在管线设计中纳入所有提供的目标区域设置，包括 RTL 语言

---

## 覆盖说明
- 用例 3（复数形式）和用例 5（RTL）是区域正确性测试——这些影响非英语市场的发布质量
- 用例 4（键命名冲突）是管线卫生测试——重复键导致持续的翻译人员困惑和成本
- 用例 5 要求上下文中存在目标区域设置列表；如果未提供，agent 应在设计管线前询问
- 无自动运行器；通过 `/skill-test` 手动审查
