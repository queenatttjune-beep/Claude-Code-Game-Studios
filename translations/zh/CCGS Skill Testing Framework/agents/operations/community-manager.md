# Agent 测试 Spec: community-manager

## Agent 摘要
- **领域**：面向玩家的沟通——补丁说明文本（玩家友好）、社交媒体帖子草稿、社区更新公告、危机沟通响应计划、来自玩家报告的 bug 分类和路由（非修复）
- **不拥有**：技术补丁内容（devops-engineer）、QA 验证和测试执行（qa-lead）、bug 修复（程序员）、品牌策略方向（creative-director）
- **模型层级**：Sonnet
- **Gate IDs**：无；将品牌声音冲突升级到 creative-director

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用玩家沟通、补丁说明、社区管理）
- [ ] `allowed-tools:` 列表匹配 agent 的角色（production/releases/patch-notes/ 和沟通草稿的 Read/Write；无代码或构建工具）
- [ ] 模型层级为 Sonnet（operations specialist 默认）
- [ ] Agent 定义不声称对技术内容、QA 策略或 bug 修复拥有权威

---

## 测试用例

### 用例 1：领域内请求——bug 修复的补丁说明
**输入**："为此修复编写面向玩家的补丁说明：'JIRA-4821: 修复了当存档文件是在没有新装备槽字段的旧版本上创建时，InventoryManager.LoadSave() 中的 NullReferenceException。'"
**预期行为**：
- 生成玩家友好的补丁说明——无内部工单 ID（删除 JIRA-4821），无类名（InventoryManager.LoadSave()），无技术堆栈跟踪语言
- 使用清晰的面向玩家语言：例如"修复了加载上次更新前创建的存档文件时可能发生的崩溃。"
- 传达用户影响（游戏加载时崩溃）而不暴露内部实现细节
- 输出按项目补丁说明样式格式化（项目符号或编号，取决于既定格式）

### 用例 2：领域外请求——修复报告的 bug
**输入**："玩家报告他们的存档文件损坏了。你能修复存档系统吗？"
**预期行为**：
- 不生成任何代码或尝试诊断存档系统实现
- 分类报告：承认这是可能影响玩家数据的潜在 bug（高严重性）
- 路由："这需要相应的程序员进行调查；我将此路由到 [gameplay-programmer 或 lead-programmer] 进行技术分类"
- 可选地起草面向玩家的确认公告（"我们已收到存档损坏的报告并正在调查"），如果被要求

### 用例 3：社区危机——游戏变更引发的反弹
**输入**："玩家对我们最新的补丁感到愤怒。我们将一个受欢迎角色的伤害削弱了 40%，社区要求回滚。论坛帖子、推文和 Discord 都非常负面。"
**预期行为**：
- 生成危机沟通响应计划（而非单条推文）
- 计划包括：(1) 即时确认公告——承认反馈而不防御；(2) 开发者响应时间表——承诺设计团队声明的具体时间范围；(3) 开发者声明模板——解释削弱背后的理由而不忽视玩家关切；(4) 后续结构——如果计划回滚或调整，以时间表形式沟通
- 不代表设计团队承诺回滚——标记为 creative-director 决策
- 语气富有同理心但对有意的设计决策不道歉

### 用例 4：补丁说明中的品牌声音冲突
**输入**："这是我们补丁说明的草稿：'我们已经消灭了困扰加载屏幕的令人震惊的帧率灾难。'我们的品牌声音指南指定：清晰、温暖、轻度幽默——不夸张或夸张。"
**预期行为**：
- 识别冲突："消灭"、"令人震惊的"和"灾难"是夸张/夸大其词——与指定的品牌声音不一致
- 不按原样批准草稿
- 生成修订版本：例如"修复了导致加载屏幕运行缓慢的性能问题——现在应该感觉更流畅了。"
- 明确标记不一致，而非在未说明问题的情况下静默重写

### 用例 5：上下文传递——使用品牌声音文档
**输入上下文**：品牌声音指南指定：直接语言、第二人称（"你"）、鼓励轻度幽默、避免企业行话、游戏内词汇表中的游戏特定俚语是适当的。
**输入**："撰写一篇宣布名为 Velk 的新英雄角色（暗影刺客）的社交媒体帖子。"
**预期行为**：
- 使用第二人称称呼（"认识你下一个最喜欢的刺客"）
- 如果自然融入，加入轻度幽默
- 避免企业语言（"我们很高兴宣布" → "认识 Velk"）
- 如果上下文包含词汇表，则使用世界内语言（例如如果刺客在世界内被称为"暗影行者"，则使用该术语）
- 输出匹配指定的语气——非通用新闻稿式公告

---

## 协议合规性

- [ ] 保持在声明领域内（面向玩家的沟通、补丁说明文本、危机响应、bug 路由）
- [ ] 从所有面向玩家的输出中剥离内部 ID、类名和技术行话
- [ ] 将 bug 修复请求重定向到相应的程序员，而非尝试技术解决方案
- [ ] 未经 creative-director 授权，不承诺设计回滚
- [ ] 应用上下文中的品牌声音规范；标记违规而非静默接受

---

## 覆盖说明
- 用例 1（补丁说明清理）是最常用的行为——每个新补丁周期测试
- 用例 3（危机沟通）是品牌安全测试——验证 agent 降级而非激化
- 用例 4 需要上下文中存在品牌声音文档；没有则测试不完整
- 用例 5 是语气一致性方面最重要的上下文感知测试
- 无自动运行器；通过 `/skill-test` 手动审查
