# Agent 测试 Spec: accessibility-specialist

## Agent 摘要
领域：输入重映射、文本缩放、色盲模式、屏幕阅读器支持和可访问性标准合规性（WCAG、平台认证）。
不拥有：整体 UX 流程设计（ux-designer）、视觉艺术风格方向（art-director）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用可访问性 / 包容性设计 / WCAG）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对 UX 流程或视觉艺术风格拥有权威

---

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "审查玩家 HUD 的可访问性。"
**预期行为：**
- 审计 HUD spec 或截图：
  - 对比度（标记任何低于 AA 的 4.5:1 或 AAA 的 7:1 的文本）
  - 颜色编码信息的替代表示（例如敌人血条仅使用颜色，无形状区别）
  - 文本大小（标记 1080p 下任何低于 16px 等效值的文本）
  - 关键状态元素的屏幕阅读器或 TTS 注释可用性
- 生成带有具体元素名称及其未满足标准的优先发现列表
- 不重新设计 HUD——产生发现供 ux-designer 和 ui-programmer 处理

### 用例 2：领域外请求——正确重定向
**输入：** "设计整体游戏流程：主菜单 → 角色选择 → 加载中 → 游戏 → 暂停 → 结果。"
**预期行为：**
- 不生成 UX 流程架构
- 明确指出整体游戏流程设计属于 `ux-designer`
- 将请求重定向到 `ux-designer`
- 可以指出一旦流程设计好，它可以从可访问性角度审查流程（例如时间限制、认知负荷）

### 用例 3：色盲模式冲突
**输入：** "为绿色弱提出的色盲模式将敌人红色血条替换为橙色，但美术调色板已使用橙色表示友方单位。"
**预期行为：**
- 识别冲突：色盲模式与已建立的友方单位调色板之间的橙色冲突
- 不单方面更改美术调色板（那属于 art-director）
- 使用描述的具体视觉重叠向 `art-director` 标记冲突
- 提出不需要调色板变更的替代区分策略（例如形状/图标覆盖、图案填充、图标学）

### 用例 4：可访问性功能所需的 UI 状态
**输入：** "库存的屏幕阅读器支持要求系统将物品名称和数量暴露为可访问文本节点。"
**预期行为：**
- 生成定义每个库存元素所需可访问文本属性的可访问性要求 spec
- 识别实现可访问文本节点需要 UI 系统更改
- 与 `ui-programmer` 协调实现所需的可访问文本节点暴露
- 不自行实现 UI 系统更改

### 用例 5：上下文传递——WCAG 2.1 目标
**输入：** 上下文中提供项目可访问性目标：WCAG 2.1 AA 合规性。请求："审查对话系统的可访问性。"
**预期行为：**
- 引用与对话相关的具体 WCAG 2.1 AA 成功标准（例如 1.4.3 对比度最小化、1.4.4 文本缩放、2.2.1 自动前进对话的时间调整）
- 使用标准中的确切标准编号和名称，而非转述
- 使用其失败的具体标准标记每个发现
- 指出哪些标准超出 AA 范围（仅 AAA），以免被错误标记为失败

## 协议合规性

- [ ] 保持在声明领域内（重映射、文本缩放、色盲模式、屏幕阅读器、标准合规性）
- [ ] 将 UX 流程设计重定向到 ux-designer，调色板决策重定向到 art-director
- [ ] 返回带有具体元素名称、对比度和标准引用的结构化发现
- [ ] 不实现 UI 更改——与 ui-programmer 协调实现
- [ ] 在提供合规目标时按编号引用具体的 WCAG 标准
- [ ] 将可访问性要求与艺术决策之间的冲突标记给 art-director

---

## 覆盖说明
- HUD 审计（用例 1）应产生可在 sprint 积压中作为可访问性 story 追踪的发现
- 色盲冲突（用例 3）确认 agent 尊重 art-director 对调色板的权威
- WCAG 标准（用例 5）验证 agent 精确地而非笼统地使用标准
