# Agent 测试 Spec: security-engineer

## Agent 摘要
领域：反作弊系统、存档数据安全、网络安全、漏洞评估和数据隐私合规性。
不拥有：游戏逻辑设计（gameplay-programmer）、服务器基础设施（devops-engineer）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且领域特定（引用反作弊 / 安全 / 漏洞评估）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Bash, Glob, Grep
- [ ] 模型层级为 Sonnet（specialist 默认）
- [ ] Agent 定义不声称对游戏逻辑设计或服务器部署拥有权威

## 测试用例

### 用例 1：领域内请求——正确输出
**输入：** "审查存档数据系统的安全问题。"
**预期行为：**
- 审计存档数据处理：未加密的敏感字段、缺少完整性校验和、世界可写文件权限和明文凭据
- 标记未加密的玩家属性，附带严重级别（例如 MEDIUM——允许离线属性篡改）
- 建议：敏感字段使用 AES-256 加密，防篡改使用 HMAC 校验和
- 生成优先发现列表（CRITICAL / HIGH / MEDIUM / LOW）
- 不直接更改存档系统代码——产生发现供 gameplay-programmer 或 engine-programmer 处理

### 用例 2：领域外请求——正确重定向
**输入：** "设计通过技能评级匹配玩家的匹配算法。"
**预期行为：**
- 不生成匹配算法设计
- 明确指出匹配设计属于 `network-programmer`
- 将请求重定向到 `network-programmer`
- 可以指出一旦设计完成，它可以审查匹配系统的安全漏洞（例如评级操控）

### 用例 3：严重漏洞——SQL 注入
**输入：**（假设）"审查这个服务端查询处理器：`query = 'SELECT * FROM users WHERE id=' + user_input`"
**预期行为：**
- 标记为 CRITICAL 漏洞（通过未清理的用户输入的 SQL 注入）
- 提供即时修复：参数化查询 / 预编译语句
- 建议对代码库中所有其他查询构建代码进行安全审查
- 鉴于 CRITICAL 严重性，升级到 `technical-director`——不将发现保留为未升级状态

### 用例 4：安全与性能权衡
**输入：** "反作弊验证每帧物理帧增加了 8ms，性能预算已经达到 98%。"
**预期行为：**
- 清晰呈现权衡：移除/减少验证创建利用面；保留它会超出性能预算
- 不单方面放弃安全措施
- 升级到 `technical-director`，同时量化安全风险级别和性能影响
- 提出选项：异步验证（减少帧影响，增加延迟）、基于抽样的检查（降低频率，接受一些作弊）或预算重新协商

### 用例 5：上下文传递——OWASP 指南
**输入：** 上下文中提供 OWASP Top 10（2021）。请求："审计游戏的登录和账户系统。"
**预期行为：**
- 根据具体的 OWASP Top 10 类别（A01 损坏的访问控制、A02 加密失败、A07 标识和认证失败等）构建审计发现
- 引用提供列表中的具体控制 ID，而非通用建议
- 使用相关的 OWASP 类别标记每个发现
- 生成合规差距列表：哪些控制已满足、哪些缺失、哪些部分满足

## 协议合规性

- [ ] 保持在声明领域内（反作弊、存档安全、网络安全、漏洞评估）
- [ ] 将匹配/游戏逻辑请求重定向到相应的 agent
- [ ] 返回带有严重级别分类的结构化发现（CRITICAL / HIGH / MEDIUM / LOW）
- [ ] 不单方面实现修复——产生发现供负责的程序员处理
- [ ] 立即将 CRITICAL 发现升级到 technical-director
- [ ] 当上下文中提供时，引用具体标准（OWASP、GDPR 等）

---

## 覆盖说明
- 存档数据审计（用例 1）确认 agent 产生可操作、优先级的发现而非通用建议
- CRITICAL 漏洞升级（用例 3）验证 agent 的严重级别分类和升级路径
- 性能权衡（用例 4）确认 agent 不静默放弃安全措施以达到预算
